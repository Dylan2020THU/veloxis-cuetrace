## 二、错误码详细排查

### 270924289 — 频率过快

排查步骤：

1. 降低请求频率，使用指数退避策略（1s → 3s → 10s）

2. 推荐改用回调模式而非轮询：通过「创建投诉通知回调地址」配置回调地址

3. 必须轮询时控制频率（建议每商户每分钟 ≤ 20 次）

4. 重试加 jitter 避免集中触发


* * *

### 270924332 — 认证信息解析失败

排查步骤：

1. `Authorization` 头必须以 `WECHATPAY2-SHA256-RSA2048 ` 开头（注意空格）

2. 必须包含五个字段： `mchid` / `nonce_str` / `signature` / `timestamp` / `serial_no`

3. 字段间用英文逗号分隔，字段值用英文双引号包裹

4. `signature` 必须是标准 Base64 编码（非 URL-Safe）

5. `timestamp` 秒级 Unix 时间戳， `nonce_str` 32 位随机字符串

6. 建议使用官方 SDK 自动生成 Authorization 头


* * *

### 269546505 — 商户证书已过期

排查步骤：

1. 登录商户平台 → 账户中心 → API 安全 → API 证书查看到期时间

2. 重新下载新证书，更新到所有服务节点

3. 建议在证书过期前 30 天监控告警


* * *

### 269546538 — 商户号不存在

排查步骤：

1. 仔细核对商户号（10 位纯数字），前后不能有空格

2. 区分调用方商户号（签名所用）与 `complainted_mchid`（被诉商户号）

3. 业务侧入参做正则校验： `^[0-9]{10}$`

4. 区分环境的商户号


* * *

### 270701078 — 无权查看当前投诉单信息

排查步骤：

1. 投诉单只能由被诉商户或其归属服务商查看

2. 检查 `complaint_id` 是否传错——可能是查询了别的商户的投诉

3. 服务商场景下确认与子商户的服务商关系已正确建立

4. 业务侧持久化「投诉单 \- 商户号」映射，确认归属后再发请求


* * *

### 268446717 — 当前查询结果数据量过大，请缩小 limit 后重试

排查步骤：

1. `limit` 范围 `[1, 50]`，不能超过 50

2. `begin_date` 与 `end_date` 跨度不能超过 30 天

3. 使用 `offset` 分页拉取，不要一次性请求大量数据

4. 同时配合 `complainted_mchid` 过滤特定子商户


* * *

### 270701074 — 无权限查询当前商户的投诉协商历史

排查步骤：

1. 与 270701078 类似，调用方与投诉单的归属关系不匹配

2. 检查 `complainted_mchid` 是否传错

3. 服务商场景下确认服务商关系仍然有效

4. 先查投诉详情，再查协商历史，避免无效请求


* * *

### 268440133 — 投诉单状态错误

排查步骤：

1. 先调「查询投诉单详情」确认当前 `complaint_state`

2. 状态机： `PENDING`（待处理）→ `PROCESSING`（处理中）→ `PROCESSED`（已处理完成，终态）

3. `PROCESSED` 状态下不能再调反馈处理完成

4. 业务侧做幂等性处理——重复的「反馈处理完成」当作成功


* * *

### 269546507 — 商户证书序列号有误

排查步骤：

1. `serial_no` 必须是当前正在使用的商户 API 证书的序列号

2. 商户平台 → 账户中心 → API 安全 → API 证书，对比序列号

3. 证书重新下载后旧的 `serial_no` 失效

4. 多实例部署时确认所有节点使用同一份证书


* * *

### 268435464 — 参数超出取值范围

排查步骤：

1. `limit` 范围 `[1, 50]`， `offset` ≥ 0

2. `begin_date` / `end_date` 格式 `yyyy-MM-DD`，跨度 ≤ 30 天

3. `complaint_id` ≤ 64 字符

4. `media_id_list` 数组长度 ≤ 4


* * *

### 269546525 — 错误的签名，导致验签失败

排查步骤：

1. 签名串拼接： `HTTP方法\nURL路径\n时间戳\n随机串\n请求体\n`

2. URL 必须用绝对路径（包含 query string，不含域名）

3. GET 请求报文主体为空字符串（仍需要 `\n`）

4. 签名算法 SHA256withRSA，使用商户 API 证书私钥

5. `signature` 必须是标准 Base64 编码（非 URL-Safe）

6. 建议使用官方 SDK 自动签名


* * *

### 270704646 — 您暂无提交商户反馈的权限

排查步骤：

1. 商户号必须已开通消费者投诉 2.0 产品权限

2. 登录商户平台 → 产品中心 → 消费者投诉，确认开通状态

3. 服务商场景下子商户也需要单独开通

4. 检查调用方是否是该投诉单的被诉商户或归属服务商

5. 商户被风控管控时投诉处理权限可能被冻结


* * *

### 269546518 — 无有效的商户证书

排查步骤：

1. 登录商户平台 → 账户中心 → API 安全 → API 证书，查看证书状态

2. 商户没有有效的 API 证书时需重新申请下载

3. 证书被撤销时需重新申请

4. 检查是否使用了错误的证书文件（PEM 文件路径错误）


* * *

### 268566836 — 插件验签时证书状态不可用

排查步骤：

1. 该错误指商户的微信支付平台证书已过期或被撤销

2. 调「获取平台证书」接口拉取最新平台证书，业务侧缓存多份覆盖轮换过渡期

3. 推荐迁移到微信支付公钥体系，规避证书轮换问题


* * *

### 268435461 — 参数非法

排查步骤：

1. 看响应体 `detail.field` 定位非法字段

2. 日期格式必须是 `yyyy-MM-DD`

3. `complainted_mchid` 必须是 10 位数字

4. `complaint_id` 长度 ≤ 64，字符集 `[a-zA-Z0-9]`

5. 回复内容不能包含敏感词 / 非法字符

6. `media_id` 不合法时检查图片上传是否过期


* * *

### 268435484 — 数据已存在

排查步骤：

1. 一个商户号只能创建一个回调地址，如需修改使用更新接口

2. 业务侧持久化「商户号 \- 回调地址 ID」映射，避免重复创建

3. 已有回调地址时先调「查询投诉通知回调地址」获取


* * *

### 268443518 — 调用下游服务失败

排查步骤：

1. 该错误为微信支付侧依赖服务异常，多为暂时性问题

2. 业务侧应当作可重试错误处理（30s → 2min → 5min 退避）

3. 大量出现时关注微信支付官方公告

4. 业务侧用消息队列异步处理避免主流程阻塞


* * *

### 268457079 — 多个状态、事件匹配的状态转移，条件均不匹配

排查步骤：

1. 业务侧通常无法直接修复，这是系统状态机错误

2. 先查「查询投诉单详情」确认当前 `complaint_state`，按状态机允许的操作发起请求

3. 投诉单状态变更存在异步延迟，刚处理过的请稍后再试

4. 反复出现需联系微信支付技术支持


* * *

### 270701064 — 提交商户反馈失败，请稍后重试

排查步骤：

1. 系统暂时性问题，使用相同参数稍后重试（10s → 30s → 1min 退避）

2. 重试前先调「查询投诉单详情」确认状态——可能首次请求已成功

3. 业务侧操作要做幂等性处理


* * *

### 270701056 — 输入参数错误

排查步骤：

1. 看响应体 `message` 字段定位具体错误原因

2. 对照官方接口文档逐一检查必填字段

3. POST 请求的 Body 字段也需要满足业务规则（如 `complainted_mchid` 必须与查询时一致）
