## 二、错误码详细排查

### 268440110 — 商户未入驻

排查步骤：

1. 登录商户平台 → 产品中心 → 我的产品，检查「微信支付分停车服务」是否在「已开通」列表

2. 未开通则提交资质申请

3. 服务商模式下子商户也需要开通该产品权限

4. 权限刚开通需等几分钟到几小时生效


* * *

### 268440144 — 车牌不存在

排查步骤：

1. `plate_number` 仅包含省份汉字 \+ 字母数字，不能含连字符 / 空格 / 中点等特殊字符

2. `plate_color` 必须从枚举里选： `BLUE` / `GREEN` / `YELLOW` / `BLACK` / `WHITE` / `LIMEGREEN`，不能传中文

3. 同一辆车的 `plate_number` 和 `plate_color` 在所有接口里必须保持一致


* * *

### 269546507 — 商户证书序列号有误

排查步骤：

1. 登录商户平台 → 账户中心 → API 安全 → API 证书管理，确认当前生效证书序列号

2. `serial_no` 与签名所用私钥必须是同一对证书

3. 近期重新申请过证书的需更新本地配置

4. 建议使用官方 SDK 自动从证书文件读取 `serial_no`


* * *

### 268440163 — 传入的 openid 不合法

排查步骤：

1. `openid` 是 AppID 维度的，不同 AppID 下用户的 openid 不同

2. 检查 `appid` 与 `openid` 是否配对

3. 服务商场景下需用 `sub_openid` 而非主商户的 openid

4. 检查 openid 是否被截断或 URL 编码污染


* * *

### 268435463 — 参数过短

排查步骤：

1. 看响应体中的 `detail.field` 字段，定位是哪个字段过短

2. 可选字段不要传空串，直接不传 key

3. 对照接口文档逐字段核对最小长度


* * *

### 268440765 — 订单不存在

排查步骤：

1. 检查 `out_trade_no` 大小写、长度、是否被截断

2. 检查查询用的商户号 `mchid` 与下单时一致

3. 「扣费受理」接口异步处理，建议下单后 2-3s 再查

4. 用 `transaction_id` 查询比 `out_trade_no` 更可靠

5. 仍查不到可用相同 `out_trade_no` 原参重发扣费受理（接口幂等）


* * *

### 268435461 — 参数非法

排查步骤：

1. 时间字段 `start_time` / `end_time` 必须符合 RFC3339 格式 `yyyy-MM-DDTHH:mm:ss+08:00`

2. 枚举值必须使用接口文档定义的取值（如 `plate_color` 不能传中文）

3. `amount.total` 必须 \> 0，单位为分

4. `trade_scene` 只能是 `PARKING`

5. `out_parking_no` / `out_trade_no` 字符集为数字、大小写字母、 `_`、 `-`、 `*`

6. 查看响应体 `detail` 字段定位具体非法字段


* * *

### 268440766 — 子商户号不匹配

排查步骤：

1. 服务商平台 → 我的特约商户，确认目标子商户号已签约

2. 子商户进件成功不等于已签约，需子商户完成开户意愿确认

3. 检查请求中的 `sub_mchid` 是否传错


* * *

### 268477397 — 子商户号（sub\_mchid）不存在

排查步骤：

1. 检查 `sub_mchid` 是否传错位

2. 商户平台核对该子商户号状态

3. 新进件子商户需等审核通过、签约完成、产品权限开通后才能使用


* * *

### 268440150 — 商户号和 appid 未绑定

排查步骤：

1. 登录商户平台 → 账户中心 → API 安全 → AppID 账号管理

2. 把要使用的 `appid` 添加到当前商户号关联列表

3. 关联生效需几分钟

4. 服务商模式下子商户的 `sub_appid` 也要与 `sub_mchid` 绑定


* * *

### 268440126 — 商户未开通垫资能力

排查步骤：

1. 该错误一般需联系微信支付运营 / BD 申请开通「微信支付分停车垫资」能力

2. 申请通过后在商户平台 → 产品中心确认能力已开通


* * *

### 269546538 — 商户号不存在

排查步骤：

1. 仔细核对 `mchid`（10 位纯数字），尤其是从配置文件 / 环境变量加载时

2. 区分测试环境与生产环境的商户号，不要混用

3. 商户号被注销后无法恢复


* * *

### 268440124 — 触发风控拦截

排查步骤：

1. 不要立即重试，避免进一步加重风控判定

2. 商户侧无法绕过用户级风控，引导用户在「微信支付分小程序」处理已有欠款 / 异常订单

3. 商户侧被频繁拦截时检查是否有异常扣费模式

4. 业务侧应记录被拦截的用户，短期内不再对其发起扣费


* * *

### 269546505 — 商户证书已过期

排查步骤：

1. 登录商户平台 → 账户中心 → API 安全 → API 证书，查看到期时间

2. 过期需重新申请并下载证书，更新到所有服务节点

3. 建议在证书过期前 30 天监控告警


* * *

### 268440164 — 入场信息不存在或已失效

排查步骤：

1. 「扣费受理」中的 `parking_info.parking_id` 应使用「创建停车入场」返回的 `id` 字段，而非商户侧的 `out_parking_no`

2. 入场记录有效期通常为 24 小时，用户出场后应及时发起扣费

3. 同一辆车多次入场时要用对应那次入场的 `parking_id`


* * *

### 268440167 — 请求不合法（entrance\_id 重复使用或参数不一致）

排查步骤：

1. 一个入场 ID 对应一次完整的入场→出场→扣费流程，扣费成功后不能再次复用

2. 如需对同一辆车再次扣费，需重新调「创建停车入场」生成新的入场 ID

3. 重试扣费时 `out_trade_no` 和所有参数必须与首次完全一致

4. 业务侧加分布式锁（按 `out_trade_no` 维度）防止并发重复请求


* * *

### 268440812 — 重复的商户入场 id 但入场信息不同

排查步骤：

1. 同一 `out_parking_no` 重试时所有参数必须完全一致

2. 如果确实是新的入场（如同一辆车不同次入场），必须使用新的 `out_parking_no`

3. 业务侧持久化首次请求的完整参数，重试时直接读取


* * *

### 268440168 — 车牌未开通支付分停车服务

排查步骤：

1. 扣费前先调「查询车牌服务开通信息」（ `GET /v3/vehicle/parking/services/find`）确认 `service_state`

2. `OUT_SERVICE` 表示未开通，引导用户通过小程序 / H5 / App 拉起开通页

3. `PAUSE` 表示用户已暂停服务，引导用户在小程序恢复


* * *

### 268440162 — 参数错误或缺失

排查步骤：

1. 看响应体 `detail.field` 定位缺失字段

2. 注意嵌套对象（如 `parking_info`）下的所有子字段都需要传

3. `notify_url` 必须是 HTTPS


* * *

### 268435462 — 参数过长

排查步骤：

1. 看响应体 `detail.field` 定位过长字段

2. 注意同一字段在不同接口的长度限制可能不同（如 `parking_name`：「创建入场」≤ 128 字符；「扣费受理」中 ≤ 32 字符）

3. 业务侧入参做长度校验，超长时截断或拒绝
