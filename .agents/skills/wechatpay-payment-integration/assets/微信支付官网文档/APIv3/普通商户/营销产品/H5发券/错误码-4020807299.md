## 二、错误码详细排查

### 272755716 — 券不在使用有效期（相对周期）

排查步骤：

1. 调「查询用户单张券详情」查看券的具体有效期

2. 商家券相对有效期由以下字段控制： `available_day_after_receive`（领取后 N 天）/ `wait_days_after_receive`（领取后等待 N 天生效）/ `available_second_after_receive`（领取后 N 秒）

3. 用户在「领取后等待天数」内券处于「未生效」状态，需等到 `wait_days_after_receive` 过后才能使用

4. H5 发券补发场景可在 URL 里传 `customize_send_time{n}` 覆盖默认领取时间


* * *

### 272757763 — 累加限额已达最大值

排查步骤：

1. 调「查询商家券批次详情」确认 `max_amount` / `max_coupons` 与已发放数量

2. 预算已达上限时新建批次或在商户平台 → 商家券 → 批次管理增加预算

3. 大型营销活动建议用多个批次轮询发放


* * *

### 272756757 — APPID 与发券商户号无绑定关系

排查步骤：

1. 登录 `send_coupon_merchant` 对应商户号 → 商户平台 → 账户中心 → API 安全 → AppID 账号管理

2. 把获取 `open_id` 时使用的 AppID 添加到关联列表，绑定后几分钟生效

3. 公众号 AppID、小程序 AppID、移动应用 AppID 需分别绑定


* * *

### 272755723 — 券不存在

排查步骤：

1. 检查 `coupon_code` 是否传错（被截断 / 编码污染）

2. 检查 `appid` \+ `openid` \+ `coupon_code` 三元组是否匹配（券归属于特定用户）

3. H5 发券失败的 `coupon_code` 无法查询


* * *

### 268578417 — 命中商品券批次日限额

排查步骤：

1. 调「查询商家券批次详情」查看 `max_amount_by_day` 与今日已发放金额

2. 等到下一个自然日 0 点自动重置

3. 如需当天继续发放，可切换到另一个批次


* * *

### 272756790 — 券已过期，不能再进行其他操作

排查步骤：

1. 调「查询用户单张券详情」查看券的 `state=EXPIRED`

2. 已过期券只能查看，不能再做任何操作

3. 引导用户领取新的有效券


* * *

### 272756745 — 券已被其他订单核销

排查步骤：

1. 业务侧需要保证单张券不会并发核销

2. 前端按钮做防重复点击

3. 后端按 `coupon_code` 加分布式锁，串行处理核销

4. 如果是异常重试导致的误报，应该把该错误码当作「核销已完成」处理


* * *

### 272755715 — 券不在使用有效期（绝对批次有效期）

排查步骤：

1. 调「查询商家券批次详情」查看 `coupon_use_rule.coupon_available_time` 中的 `available_begin_time` / `available_end_time`

2. 当前时间在 `available_begin_time` 之前需等批次激活

3. 当前时间超过 `available_end_time` 则批次已永久过期，无法核销

4. 部分批次配置了「每周可用日」 `available_week`，需在大窗口内同时满足小窗口


* * *

### 269546505 — 商户证书已过期

排查步骤：

1. H5 发券本身用 V2 签名不需要证书，但配套的查询接口需要证书

2. 登录商户平台 → 账户中心 → API 安全 → API 证书查看到期时间

3. 重新下载新证书，更新到所有服务节点


* * *

### 272758279 — 用户 openid 与 appid 不匹配

排查步骤：

1. `openid` 是 AppID 维度的，不同 AppID 下不同

2. 检查获取 `openid` 时用的 AppID 与发券时使用的发券商户号绑定的 AppID 是否一致

3. 公众号 / 小程序 / 移动应用 AppID 不能混用同一个 openid


* * *

### 269546538 — 商户号不存在

排查步骤：

1. 仔细核对商户号（10 位纯数字），前后不能有空格

2. URL 中 `send_coupon_merchant` ≤ 15 字符（实际为 10 位数字）

3. 区分测试 / 生产环境的商户号


* * *

### 268435462 — 参数过长

排查步骤：

1. 看响应错误页中 `detail` 定位过长字段

2. URL 总长度不超过 2048 字节（浏览器兜底）

3. 关键长度限制：

   - `stock_id` ≤ 20

   - `out_request_no` ≤ 128，字符集 `[a-zA-Z0-9｜_*-]`

   - `open_id` ≤ 128

   - `coupon_code` ≤ 128

* * *

### 270924345 — Content-Type 不正确

排查步骤：

1. H5 发券是 GET 重定向场景，正常不需要 Content-Type

2. 配套的 V3 商家券管理接口（POST）必须 `Content-Type: application/json`

3. HTTP 客户端默认可能是 `application/x-www-form-urlencoded`，需显式设置


* * *

### 272756781 — 券当前是可使用的

排查步骤：

1. 这是查询券状态时返回的状态码，意味着券可以正常核销

2. 业务侧应该把它当作「OK」处理，而非错误


* * *

### 272756769 — 券已被删除

排查步骤：

1. 调「查询用户单张券详情」确认券的 `state=DELETED`

2. 已删除的券不可恢复，引导用户重新领取

3. 商户侧误删时需新建批次重发


* * *

### 272756746 — 券已被其他订单核销锁定

排查步骤：

1. 商家券核销采用「锁定 → 确认」两步走流程，锁定期间券不可被其他订单使用

2. 业务侧按 `coupon_code` 加分布式锁，串行处理

3. 锁定通常有过期时间（如 5 分钟），过期后券自动释放


* * *

### 268574507 — 用户未开通会员

排查步骤：

1. 调「查询商家券批次详情」确认 `stock_send_rule.member_only` 与 `coupon_use_rule.member_only`

2. 引导用户先注册成为商户会员，再领券

3. 业务侧发券前先校验用户会员状态，非会员直接拒绝


* * *

### 272758283 — 查询商户 id 错误

排查步骤：

1. 核对 `send_coupon_merchant` 与系统中实际配置的发券商户号是否一致

2. 区分批次创建商户号（ `belong_merchant`）与发券商户号（ `send_coupon_merchant`）

3. 跨商户发券场景下， `send_coupon_merchant` 必须是制券商户号、券归属商户号、或有合作授权关系的商户号


* * *

### 272755255 — 校验商户号与 AppID 关联关系失败

排查步骤：

1. 商户平台 → AppID 账号管理，确认绑定关系存在

2. 服务商场景下注意是「服务商商户号 \- 服务商 AppID」还是「子商户号 - 子 AppID」绑定


* * *

### 268435461 — 参数非法（H5 领券域名参数错误）

排查步骤：

1. H5 领券 URL 末尾必须严格包含 `#wechat_pay&wechat_redirect`（注意大小写）

2. 业务侧拼接 URL 时把这部分作为常量固定追加

3. URL 编码时注意 `#` 不要被编码（ `#` 不能用 `%23`）

4. 重定向链上的中间页要保留 URL 末尾的 fragment
