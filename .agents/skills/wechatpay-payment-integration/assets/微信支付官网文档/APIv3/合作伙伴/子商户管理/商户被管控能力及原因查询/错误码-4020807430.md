## 二、错误码详细排查

### 268541152 — 查询管控时商户关系非受理模式

排查步骤：

1. 检查请求头 `Authorization` 中签名使用的 `mchid` 是否真正归属该 `sub_mchid` 的普通服务商号

2. 确认 `sub_mchid` 是通过特约商户进件（ `/v3/applyment4sub/applyment/`）进件成功的（电商二级商户不适用本接口）

3. 业务侧维护「服务商号 → 旗下特约商户号」白名单，调用前先做归属校验

4. 多 SP 商户号场景，业务侧做好「子商户 → 提交服务商号」映射

5. 跨业务体系隔离调用：电商二级商户不要走本接口


* * *

### 268541151 — 商户身份非法

排查步骤：

1. 登录商户平台 → 账户中心 → 商户信息查看商户类型

2. 本接口仅支持普通服务商

3. 不支持：普通商户（特约商户）、电商平台服务商、从业机构（银行 / 支付机构）、渠道商

4. 若主体确实是普通服务商但报错，检查是否签名使用了错误的 `mchid`


* * *

### 268590530 — 小游戏商户资料互通接口已下线

排查步骤：

1. 该错误指对应的小游戏商户资料互通历史通道已下线

2. 检查 `sub_mchid` 是不是历史小游戏商户号，如果是应当从配置中移除

3. 业务侧识别该错误码后直接丢弃该 `sub_mchid`，不应重试

4. 定期清理已下线接口对应的子商户配置


* * *

### 269795842 — 输入参数错误

排查步骤：

1. 看响应体 `message` 与 `detail.field` 定位具体错误

2. 业务侧入参做正则校验： `^[0-9]{8,10}$`

3. 标准 URL 模板： `GET /v3/mch-operation-manage/merchant-limitations/sub-mchid/{sub_mchid}`

4. URL 拼接前先做非空校验，避免空字符串污染 path 形成双斜杠

5. 不要在 path 上追加 query 参数


* * *

### 270924346 — 签名信息非法

排查步骤：

1. `Authorization` 头必须以 `WECHATPAY2-SHA256-RSA2048 ` 开头（注意空格）

2. 必须包含五个字段： `mchid` / `nonce_str` / `signature` / `timestamp` / `serial_no`

3. 字段间用英文逗号分隔，字段值用英文双引号包裹

4. `signature` 必须是标准 Base64 编码（非 URL-Safe）

5. `serial_no` 是当前商户 API 证书的序列号（不是平台证书）

6. 本接口是 GET 方法，签名串首行是 `GET`，body 用空字符串，URL 必须包含完整 path（含 `sub_mchid`）

7. 建议使用官方 SDK 自动生成 Authorization
