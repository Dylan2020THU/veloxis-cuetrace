## 二、错误码详细排查

### 268511396 — 没有进行中的处罚

排查步骤：

1. 业务侧应先通过「商户被管控能力及原因查询」接口判断商户是否真的被管控，再决定是否发起核实

2. 结合「商户平台处置通知」事件驱动：收到处置通知 → 落库管控状态 → 发起核实，而不是定时全量扫描

3. 业务侧维护「 `sub_mchid` → 管控状态 / 最近一次核实单状态」缓存

4. 出现该错误码时应当作「商户当前无需核实」的正常业务态，不要重试 / 不要告警

5. 若有多个服务实例，加分布式锁保证单商户单核实串行化


* * *

### 268511401 — 没有此单据

排查步骤：

1. 业务侧打印请求 URL，确认 path 段完整

2. 提交时发起接口返回的 `verification_id` 立即本地落库，与 `sub_mchid` 严格绑定

3. 查询时严格按「同一对」 `sub_mchid` \+ `verification_id` 拼接

4. 跨服务商查询不被允许（会同时报 403 NO\_AUTH）

5. 404 视为「单据不存在」正常业务态，不要重试


* * *

### 268511394 — 商户非电商子

排查步骤：

1. 本接口仅支持电商二级商户（通过 `POST /v3/ecommerce/applyments/` 进件成功的子商户）

2. 普通服务商通过特约商户进件（ `/v3/applyment4sub/`）创建的商户不能用本接口

3. 业务侧记录每个 `sub_mchid` 的进件类型标签（ `ECOMMERCE` / `NORMAL`）

4. 普通服务商的子商户需走其他合规处理流程


* * *

### 268511316 — 其它失败

排查步骤：

1. 看响应体 `message` 是否带有更具体的子原因

2. 后端依赖服务瞬时异常，使用相同参数稍后重试（5s → 30s → 1min 指数退避）

3. 重试前先调查询接口确认是否已经创建成功（避免重复创建）

4. 切换备域名重试（ `api2.mch.weixin.qq.com`）

5. 长时间持续失败时联系微信支付技术支持


* * *

### 268511311 — 子商户非法

排查步骤：

1. 仔细核对 `sub_mchid`——纯数字，前后不能有空格

2. 业务侧入参做正则校验： `^[0-9]{8,10}$`

3. 检查 `sub_mchid` 进件状态是否为 `FINISHED`

4. 不要传 null / 空字符串到 API

5. 维护「服务商号 → 旗下子商户号」白名单，调用前先做归属校验


* * *

### 268511395 — 商户非服务商子商户

排查步骤：

1. 检查请求头 `Authorization` 中签名使用的 `mchid` 是否真正归属该子商户的平台商户号

2. 翻进件时的日志，确认该 `sub_mchid` 是通过当前服务商进件的

3. 多 SP 商户号场景，业务侧做好「子商户 → 提交平台商户号」映射

4. 不允许跨服务商访问


* * *

### 268435461 — 参数非法

排查步骤：

1. 看响应体 `message` 与 `detail.field` 定位具体错误字段

2. 发起接口：POST 请求 body 仅包含 `{"sub_mchid": "xxx"}`

3. 查询接口：GET 请求，参数在 path，body 应为空

4. `sub_mchid` / `verification_id` 长度均不超过 32 字符

5. 不要传非接口规范字段
