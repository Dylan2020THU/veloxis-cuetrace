第一段：认证架构与数据边界

核心原则：账号是唯一主体；手机号、账号名、微信只是可选凭证；登录会话才代表当前设备已登录。

手机号 + 验证码
手机号/账号 + 密码
已绑定微信
→
accountAuth

解析到唯一 accountId
→
auth_sessions
签发随机会话令牌
服务端仅保存令牌哈希
↓
所有受保护业务云函数
requireSession → accountId → users.roles
不再用 OPENID 直接推断当前登录账号
数据关系
accounts · 账号主体
不可变随机 accountId

状态、可选密码哈希、会话撤销版本。账号名和手机号变化都不会改变主体 ID。

account_names · 0 或 1
唯一规范化账号名 → accountId
phone_bindings · 0 或 1
唯一手机号 HMAC → accountId；不以明文手机号作为文档 ID
wechat_bindings · 0 或 1
可信 OPENID 哈希 → accountId；只有明确同意才写入
users · 严格 1 对 1
accountId → 资料、member/coach/shop 角色
临时安全数据
sms_codes
按手机号 HMAC 与用途保存短期挑战；不再依赖 OPENID 才能验证。
auth_sessions · 1 对多
每台设备独立会话，可单独撤销；密码变更或“退出全部设备”统一撤销。
OPENID 的新职责
仅用于“微信登录”“用户主动绑定微信”和短信风控辅助；它不再是手机号/账号密码登录后的业务账号主键。
