# 微信身份与业务账号 1:1 绑定设计

## 状态与授权

- 日期：2026-07-10
- 状态：已批准，可直接进入实施计划与执行
- 用户明确选择：
  - 一个微信身份只能绑定一个业务账号，一个业务账号也只能绑定一个微信身份。
  - 当前没有需要迁移的生产账号或生产数据，可以重建云端账号体系。
  - 后续选项按本文推荐方案自主决定，保留完整过程供事后 review。
- 本设计批准仅覆盖本文范围；不授权删除任何仓库文件。

## 背景与根因

登录页当前有微信图标和点击事件，但所谓“微信绑定”只在本机写入 `dc_wechat_bindings` 与 `dc_accounts[].wechatBound`。绑定记录不包含 `OPENID/UNIONID`，后续登录也不读取它；唯一读取方是账号安全页，用于显示“已绑定”。

云端 `login` 能通过 `cloud.getWXContext().OPENID` 识别当前微信用户，但普通账号名没有写入 `users`，客户端提交的 `roles` 反而会被合并进云端用户，形成越权风险。应用启动又使用有写副作用的 `login` 作为云探针，失败时静默回退到 mock 登录，进一步掩盖 AppID、云环境或云函数部署问题。

因此缺失的不是一个单独的 `wx.login()` 调用，而是：

1. 云端业务账号主体；
2. 可信的账号凭据校验；
3. 业务账号与微信 `OPENID` 的双向唯一映射；
4. 绑定记录在后续免密登录中的消费；
5. 服务端角色授权边界。

## 目标

1. 建立云端业务账号，并彻底停止使用本地明文账号作为鉴权来源。
2. 严格保证 `1 个 OPENID ↔ 1 个业务账号`。
3. 首次注册或账号验证成功后绑定当前微信；以后点击微信图标可跨本地缓存、跨设备免密恢复同一账号。
4. 微信身份只取自 `cloud.getWXContext()`，不接受客户端传入 `openid`。
5. 密码只在云端保存不可逆哈希，不保存或返回明文密码。
6. 业务角色只由服务端注册和审批链路授予；客户端只能请求“本次进入哪个已授权角色”。
7. 云不可用时认证失败关闭，不再伪造本地登录成功。
8. 保持现有登录页整体布局和“认证后选择身份”的交互结构。

## 非目标

1. 不迁移 `dc_accounts`、`dc_wechat_bindings` 或其他测试账号数据。
2. 不支持一个微信绑定多个账号，或一个账号绑定多个微信。
3. 不在本次实现解绑、换绑、密码找回、密码修改和账号合并。
4. 不引入自建 JWT、refresh token 或独立 HTTP 认证服务器；继续使用微信云开发可信调用上下文。
5. 不获取用户可见的“微信号”、昵称或头像作为身份凭据。
6. 不重做登录页视觉设计。
7. 不自动删除云数据库旧集合或仓库文件；部署前的数据清理由人工决定。

## 方案比较

### 方案一：独立 `accountAuth` 云函数 + 收紧现有 `login`（采用）

新增一个职责集中的 `accountAuth` 云函数，处理注册、密码验证、微信免密、绑定状态和无副作用探针。现有 `login` 只负责在绑定关系成立后选择角色、更新当前角色并返回用户资料。

优点：

- 认证、绑定和角色选择边界清晰；
- 现有业务云函数与 `users` 数据模型改动较小；
- 可集中实现密码哈希、事务与统一错误码；
- 测试可以围绕一个认证入口覆盖完整状态机。

代价：需要新增一个云函数和两个集合，并修改登录页、数据服务、云探针和现有 `login`。

### 方案二：按注册、密码登录、微信登录拆成多个云函数

优点是每个云函数职责更窄；缺点是密码与绑定规则会分散，容易出现校验和错误码漂移，并增加部署与测试数量。当前规模不需要这层拆分，因此不采用。

### 方案三：只扩展现有 `login` 或继续使用本地账号

只扩展 `login` 会继续把云连通性探针、开户、凭据验证、绑定、角色选择混在一起；保留本地账号则服务端无法可信验证密码，也无法保证跨设备和双向唯一。两者都不能解决根因，因此不采用。

## 总体架构

### 1. 小程序登录页

登录页仅收集用户输入和展示状态，不再读取、写入或校验 `dc_accounts`/`dc_wechat_bindings`。

- 账号密码登录：调用 `accountAuth.passwordLogin`。
- 注册：调用 `accountAuth.register`，成功即绑定当前微信并进入身份选择。
- 微信图标：调用 `accountAuth.wechatLogin`；已绑定则直接进入身份选择，未绑定才展示绑定/注册提示。
- 手机号登录：短信校验成功后只允许恢复当前微信已绑定的账号；未绑定微信不得据此静默开户或抢绑账号。
- 身份选择：只消费服务端返回的 `roles`。

### 2. `miniprogram/services/data.js`

新增统一的 `accountAuth(action, payload)` 封装和语义化方法：

- `registerAccount`
- `loginWithPassword`
- `loginWithWechat`
- `getAccountSecurity`
- `probeAuthCloud`

认证方法在 `cloudReady=false` 时统一抛出 `CLOUD_NOT_READY`，不得走 mock 成功分支。

### 3. `accountAuth` 云函数

`accountAuth` 根据 `event.action` 分派以下操作：

- `probe`：只返回云函数和微信上下文可用状态，不读写账号、绑定或用户数据。
- `register`：校验账号规则和密码强度，在事务内创建账号、微信绑定和默认会员用户。
- `passwordLogin`：验证云端密码；账号未绑定时绑定当前微信，已绑定当前微信时直接成功，绑定其他微信时拒绝。
- `wechatLogin`：仅凭可信 `OPENID` 解析绑定和账号，返回账号与服务端角色；未绑定时零写入返回 `WECHAT_NOT_BOUND`。
- `status`：返回当前微信绑定的账号、安全状态和服务端角色，不返回密码哈希或原始绑定密钥。

### 4. 收紧后的 `login` 云函数

`login` 只接受 `event.role` 作为本次进入角色：

1. 根据当前 `OPENID` 解析 `wechat_bindings` 和 `accounts`；未绑定返回 `ACCOUNT_NOT_BOUND`。
2. 从 `users.roles` 读取服务端角色；不再读取 `event.roles`。
3. `event.role` 不在服务端角色内时返回 `ROLE_NOT_ALLOWED`。
4. 成功时只更新 `currentRole`、`role` 与 `updatedAt`，不通过登录请求授予新角色。
5. 不再根据普通登录请求中的 `loginName` 创建管理员。

`users.roles` 经本次收口后成为服务端业务角色账本：

- 注册只授予 `member`；
- 教练角色由店主审批教练申请后授予；
- 店主角色由管理员审批店主申请后授予；
- 管理员继续使用独立 `admins` 权限层，不混入业务 `roles`。

### 5. 账号安全页

账号安全页改为读取 `accountAuth.status`：

- 账号名来自云端账号；
- 密码只显示“已设置”，不读取密码内容；
- 微信绑定状态来自真实绑定记录；
- 手机号来自服务端已验证资料。

## 数据模型

### `accounts`

文档 ID：`sha256('account:' + accountNormalized)`，以确定性 ID 保证账号名唯一。

字段：

```js
{
  _id,
  _openid,             // 当前唯一绑定的微信 OPENID
  account,             // 展示账号名
  accountNormalized,   // 小写规范化账号名
  passwordAlgorithm,   // 'scrypt-v1'
  passwordSalt,        // 随机盐，hex
  passwordHash,        // scrypt 结果，hex
  status,              // 'active'
  createdAt,
  updatedAt,
  boundAt
}
```

禁止保存 `password` 明文字段，禁止客户端直接读取该集合。

### `wechat_bindings`

文档 ID：`sha256('wechat:' + OPENID)`，以确定性 ID 保证一个微信只有一个绑定。

字段：

```js
{
  _id,
  _openid,
  accountId,
  account,
  unionidHash,         // 仅在上下文存在 UNIONID 时保存哈希；不作为当前主键
  boundAt,
  updatedAt
}
```

`accountId` 与 `accounts._id` 一一对应。注册和首次绑定必须在同一事务中检查并写入两个确定性文档；任何一侧已有不同绑定都必须整体失败。

### `users`

继续保存业务资料和角色。新注册默认：

```js
{
  _id,                  // 新注册使用 bindingId，便于与账号/绑定在同一事务创建
  _openid,
  roles: ['member'],
  currentRole: 'member',
  role: 'member',
  nickname: '',
  avatar: '',
  createdAt,
  updatedAt
}
```

`accounts` 不重复保存角色，避免账号凭据与业务授权出现双账本。

## 核心流程

### 启动探针

1. 小程序初始化微信云开发。
2. 调用 `accountAuth({ action: 'probe' })`。
3. 调用成功即 `cloudReady=true`，无论当前微信是否已绑定账号。
4. 探针不得创建 `users`、账号或绑定记录。

### 注册并绑定

1. 客户端校验协议勾选、账号格式、密码长度与两次密码一致。
2. 调用 `accountAuth.register`。
3. 服务端规范化账号并生成确定性 `accountId/bindingId`。
4. 事务检查：账号不存在、当前微信未绑定其他账号。
5. 使用随机盐和 `scrypt` 生成密码哈希。
6. 事务创建 `accounts`、`wechat_bindings` 和默认 `users`。
7. 返回账号和 `roles:['member']`，客户端直接进入身份选择。

### 账号密码登录或首次绑定

1. 服务端按规范化账号读取确定性账号文档。
2. 使用 `crypto.timingSafeEqual` 比较 `scrypt` 结果；账号不存在和密码错误均返回 `INVALID_CREDENTIALS`。
3. 若账号与当前微信已双向绑定，返回账号和角色。
4. 若账号尚未绑定且当前微信也未绑定，在事务内建立双向绑定。
5. 若任一侧已绑定其他对象，返回明确冲突码且零写入。

### 微信免密登录

1. 用户点击微信图标。
2. 客户端直接调用 `accountAuth.wechatLogin`，不提交账号、密码或 `openid`。
3. 服务端用当前 `OPENID` 的确定性绑定 ID 查找账号。
4. 成功后返回账号和服务端角色并进入身份选择。
5. 未绑定返回 `WECHAT_NOT_BOUND`，页面切换到绑定/注册引导。

### 手机号登录

1. 删除“手机号必须存在于本地账号”的前置判断。
2. 短信验证码仍绑定当前可信 `OPENID`。
3. `verifySmsCode` 成功后必须确认当前 `OPENID` 已有业务账号绑定，才可更新该用户的已验证手机号并返回账号角色。
4. 未绑定微信返回 `WECHAT_NOT_BOUND`，不得创建孤立 `users` 或自动占用其他账号。
5. 手机号在本次范围内是已绑定账号的附加验证方式，不提供跨微信换绑或账号找回。

### 身份选择

1. 选择器展示服务端返回的角色集合。
2. 客户端调用 `login({ role })`，不再传 `roles/loginName`。
3. 服务端再次检查角色是否仍获授权，避免客户端缓存或篡改。
4. 成功后进入对应端口。

## 冲突与错误码

| 错误码 | 含义 | 客户端处理 |
|---|---|---|
| `CLOUD_NOT_READY` | 云服务未连接 | 停留登录页，提示检查网络或服务配置 |
| `INVALID_INPUT` | 账号或密码格式不合规 | 展示具体输入提示 |
| `ACCOUNT_EXISTS` | 注册账号已存在 | 提示改用登录 |
| `INVALID_CREDENTIALS` | 账号不存在或密码错误 | 使用统一文案，避免枚举账号 |
| `WECHAT_NOT_BOUND` | 当前微信未绑定账号 | 展示绑定或注册入口 |
| `ACCOUNT_NOT_BOUND` | 业务登录时未找到完整账号绑定 | 返回登录页，不创建临时用户 |
| `WECHAT_ALREADY_BOUND` | 当前微信已绑定其他账号 | 拒绝操作，不覆盖绑定 |
| `ACCOUNT_ALREADY_BOUND` | 账号已绑定其他微信 | 拒绝操作，提示联系支持 |
| `ACCOUNT_DISABLED` | 账号被停用 | 拒绝登录 |
| `ROLE_NOT_ALLOWED` | 请求角色未获服务端授权 | 返回身份选择并刷新角色 |
| `AUTH_INTERNAL_ERROR` | 数据库或事务失败 | 不返回成功，不回退 mock |

## 角色与管理员安全收口

1. 删除 `login` 对客户端 `event.roles` 的读取和合并。
2. 删除普通 `login` 中根据 `loginName='admin_zhx'` 写入 `admins` 的旁路。
3. `saveUserProfile` 只能把 `currentRole` 设为现有 `users.roles` 中的角色，不能自行授予角色。
4. `reviewCoachBindingApplication` 和 `reviewShopApplication` 是 coach/shop 的授权入口；审批通过时确保对应 `users` 存在并更新角色。
5. `saveShopProfile` 只能在服务端已批准的前提下保持 shop 角色一致。
6. `addShopCoach` 只允许关联已经拥有 coach 角色的用户，不能借关联动作授予角色。
7. 管理员仍只能通过 `adminLogin` 的服务端凭据校验建立 `admins` 记录；普通账号注册禁止使用保留管理员账号名。
8. `adminLogin` 也必须分别检查“管理员账号是否已绑定其他 OPENID”和“当前 OPENID 是否已绑定其他管理员账号”，不得通过 upsert 形成多对多管理员绑定。

## 本地状态与兼容策略

- `dc_accounts` 与 `dc_wechat_bindings` 不迁移、不再读取，也不作为回退。

- `users.role` 的旧单值记录只用于一次性兼容推导：`coach → member+coach`、`shop → shop`、其他 → `member`；之后角色只由服务端流程维护。
- 当前没有生产数据，因此不提供自动数据库迁移或删除脚本。

## 验收标准

1. 新用户注册后，云端立即形成唯一账号与当前微信绑定，并直接进入身份选择。

4. 第二个微信不能登录或绑定已占用账号；已绑定微信不能再绑定第二个账号。

6. 客户端不能通过 `roles`、`role` 或 `loginName` 自行获得 coach、shop 或 admin 权限。
7. 云服务不可用或数据库失败时，登录明确失败且不进入业务页面。
8. 账号安全页展示的账号、手机号和微信绑定状态均来自云端真实数据。
9. 全量自动化测试、语法检查和 diff 检查通过。
10. 真机验证项有明确结果；若受外部环境阻塞，则作为唯一待人工验收项列出。

## 过程决策记录

| 决策 | 选择 | 理由 |
|---|---|---|
| 绑定基数 | 严格双向 1:1 | 防止账号共享、抢绑和身份歧义 |
| 数据迁移 | 不迁移测试账号 | 用户确认无生产数据，避免为不可信本地明文记录设计迁移 |
| 微信身份来源 | 云函数 `getWXContext()` | 服务端可信，不接受客户端伪造 OPENID |
| 认证组织 | 独立 `accountAuth` | 与角色选择、业务登录和云探针解耦 |
| 密码方案 | `scrypt` + 随机盐 + 常量时间比较 | Node 原生、无需新增依赖、避免明文 |
| 唯一性 | 确定性文档 ID + 数据库事务 | 同时约束账号方向和微信方向，避免并发双绑 |
| 角色来源 | 收口后的 `users.roles` | 复用现有审批链路，避免凭据表和业务权限双账本 |
| 云失败策略 | 失败关闭 | 认证不能回退到 mock 假成功 |
| 微信登录 API | 不额外调用 `wx.login` | 当前架构已由微信云函数上下文提供可信 OPENID |
| 手机号登录 | 仅作为已绑定账号的附加验证 | 不允许手机号绕过微信 1:1 或静默创建孤立用户 |
| UI 范围 | 保持布局，只改状态与调用 | 聚焦根因，避免无关视觉重构 |
