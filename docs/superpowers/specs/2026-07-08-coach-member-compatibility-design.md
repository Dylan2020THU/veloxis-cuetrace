# 教练账号向下兼容球员账号设计

## 背景

当前小程序有球员、教练、店主三个端口。教练端和球员端都属于个人用户能力，但现有登录、注册和部分本地状态仍容易把“账号资格”和“当前进入端口”混在一起。这样会产生一个核心问题：一个教练长期执教后，如果后续不再做教练，他进入球员端时不应该丢失历史训练、教学课时、头像、昵称、社区等个人数据。

本设计明确：教练账号向下兼容球员账号。教练是球员的增强身份，不是另一个独立账号。

## 目标

1. 教练天然拥有球员身份，可直接登录球员端。
2. 教练退役或停用教练资格后，仍能以球员身份查看自己的历史数据。
3. 球员端杆迹合并展示普通训练和本人作为教练产生的教学课时。
4. 店主保持经营主体定位，不向下兼容球员身份。
5. 历史同名球员/教练账号合并为一个个人账号，保留双方数据。

## 非目标

1. 不把教学课时复制写入 `training_sessions`，避免双写不一致。
2. 不新增“教练退役迁移”流程，因为同一用户主体天然保留历史。
3. 不让店主自动拥有球员身份。
4. 不改社区、约球、预约等业务的主体数据结构，除非它们读取当前身份时需要跟随新模型做最小兼容。

## 身份模型

用户主体按 `openid/accountId` 归集。身份资格和当前端口分离：

- `roles`: 用户拥有的身份资格。
- `currentRole`: 用户本次进入的小程序端口。

个人账号规则：

- 球员注册后：`roles: ['member']`
- 教练注册后：`roles: ['member', 'coach']`
- 店主注册后：`roles: ['shop']`

旧数据兼容：

- 旧 `role: 'member'` 视为 `roles: ['member']`
- 旧 `role: 'coach'` 视为 `roles: ['member', 'coach']`
- 旧 `role: 'shop'` 视为 `roles: ['shop']`

`users.role` 不再作为“唯一身份”使用。短期兼容期内可以继续保留 `role` 字段，但它只能作为旧数据兜底或最近一次端口值，不作为权限判断的唯一依据。

## 注册流程

球员注册：

- 创建个人账号。
- 写入 `roles: ['member']`。
- 默认进入球员端。

教练注册：

- 创建个人账号。
- 写入 `roles: ['member', 'coach']`。
- 默认进入教练端。
- 同时允许后续直接进入球员端。

店主注册：

- 创建经营账号。
- 写入 `roles: ['shop']`。
- 继续走店主资质审核网关。

账号冲突处理：

- 已是教练的账号再次注册球员时，不新建账号，提示“该账号已具备球员身份，可直接登录球员端”。
- 已是球员的账号注册教练时，不新建账号，改为给原账号追加 `coach` 资格。
- 历史上同一账号名或手机号已分别注册球员和教练时，合并为一个个人账号，保留 `roles: ['member', 'coach']`。
- 合并时账号密码、手机号、微信绑定优先取更新较新的记录；冲突时保留现有可登录记录，不自动删除旧记录。

## 登录流程

登录页仍允许先选择端口：球员、教练、店主。

选择球员端：

- `roles` 包含 `member` 即允许登录。
- 教练账号一定可以登录球员端。
- 店主账号不能登录球员端。
- 登录成功后 `currentRole = 'member'`。

选择教练端：

- `roles` 包含 `coach` 才允许登录。
- 纯球员账号不能直接进入教练端，提示“该账号尚未开通教练身份”。
- 登录成功后 `currentRole = 'coach'`。

选择店主端：

- `roles` 包含 `shop` 才允许进入。
- 店主仍走资质审核状态判断。
- 登录成功后 `currentRole = 'shop'`。

前端底部栏、首页、端口权限判断看 `currentRole`。头像、昵称、账号安全、训练历史、社区作者等个人主体信息看同一个用户主体。

## 杆迹数据合并

球员端杆迹读取两类数据：

- 普通训练：`training_sessions`
- 教学课时：`coach_lessons`

当用户查看自己的球员端杆迹时：

- 合并 `training_sessions` 中 `_openid = 当前用户` 的记录。
- 合并 `coach_lessons` 中 `coachOpenid = 当前用户` 的记录。
- 返回条目增加 `kind: 'personal' | 'coach'`。

当教练查看学员或店主查看会员时：

- 只查看目标用户的普通训练。
- 不混入目标用户本人作为教练产生的教学课时。

热力图规则：

- 只有普通训练：蓝色。
- 只有教学课时：金色。
- 两者都有：优先显示金色。
- 累计时长、累计天数、连续打卡天数默认合并计算。

明细规则：

- 明细区增加筛选：全部、普通训练、教学课时。
- 普通训练显示球厅、开始时间、训练时长、已核验/未核验。
- 教学课时显示球厅、学员昵称、课时时长、金底“教学课时”标签。
- 教学课时可保留金额字段，但球员端默认不突出展示结算金额。

## 需要调整的模块

云函数：

- `login`: 返回 `roles` 和 `currentRole`，并兼容旧 `role`。
- `getUserProfile`: 返回用户主体资料和身份资格，不用单值 `role` 覆盖资格。
- `saveUserProfile`: 保存昵称、头像、手机号、隐私设置等主体资料，不把端口切换写成唯一身份。
- `getHeatmap`: 自查时合并 `training_sessions` 和 `coach_lessons`；查别人时只查普通训练。
- `getDayDetail`: 自查时合并两类明细并返回 `kind`；查别人时只查普通训练。

前端：

- `services/data.js`: `login(role)` 改为请求进入某端口；本地保存 `roles` 和 `currentRole`。
- `pages/login`: 注册和登录校验按 `roles` 判断资格；同名球员/教练历史记录合并。
- 杆迹相关页面：热力图支持混合来源；明细增加筛选和教学课时标签。
- 自定义底部栏：继续按当前端口渲染，不按账号资格渲染。

测试：

- 教练账号可登录球员端。
- 纯球员账号不能登录教练端。
- 店主账号不能登录球员端。
- 旧 `role: 'coach'` 自动兼容为 `roles: ['member', 'coach']`。
- 教练自己的球员端杆迹合并教学课时。
- 教练查看学员时不混入该学员自己的教学课时。
- 店主查看会员时不混入教学课时。
- 历史同名球员/教练账号合并后保留双方数据。

## 验收标准

1. 一个教练账号选择球员端登录后，可以进入球员端首页。
2. 同一教练账号在球员端看到普通训练和教学课时合并后的杆迹统计。
3. 明细列表可以筛选全部、普通训练、教学课时。
4. 教练资格停用后，用户仍能以球员身份登录并看到历史数据。
5. 纯球员账号进入教练端时收到明确提示，不能直接进入。
6. 店主账号不能进入球员端。
7. 旧数据不需要人工迁移即可按兼容规则读取。
