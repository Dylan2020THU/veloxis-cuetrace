# 手机号、账号密码与微信独立登录体系设计

## 状态

- 日期：2026-07-15
- 状态：交互设计已逐段批准并完成独立自审，待张总书面复核
- 范围：小程序用户账号认证、登录会话、手机号与微信首次绑定
- 数据前提：现有账号、微信绑定与用户主体均为测试数据，不做生产数据迁移；部署前仍需盘点所有引用旧用户 ID 的业务测试数据

## 背景

当前实现把业务账号与微信 `OPENID` 严格绑定：注册账号会自动绑定当前微信，手机号验证码也只能登录当前微信已经绑定的账号。`verifySmsCode` 通过 `OPENID` 读取 `wechat_bindings`、`accounts` 和 `users`，因此不能满足以下真实需求：

1. 新用户输入手机号和验证码后直接创建账号，不必先设置账号名或密码。
2. 已有账号在“账号与安全”中绑定手机号后，可以使用同一个账号密码配合手机号登录，也可以使用短信验证码登录。
3. 手机号登录和账号密码登录不得自动绑定当前微信。
4. 只有用户主动绑定微信，或从微信登录入口完成手机号验证后明确确认，才建立微信绑定。
5. 手机号登录不依赖微信绑定，因此业务授权不能继续把 `OPENID` 等同于当前登录账号。

真机首次获取短信验证码时还暴露出一个现有缺陷：CloudBase 对不存在文档执行 `doc.get()` 会抛出“document does not exist”，而当前 `sendSmsCode` 把首次不存在误当作可直接读取的空结果，导致短信供应商 API 尚未调用就失败。本设计将该缺陷纳入认证重构与回归测试。

## 目标

1. 建立不可变的内部账号主体，账号名、手机号、密码和微信均作为可选认证凭证。
2. 一个手机号全平台只能绑定一个账号，一个账号最多绑定一个手机号。
3. 一个微信身份最多绑定一个账号，一个账号最多绑定一个微信身份。
4. 手机号首次验证码登录时自动创建基础球员账号，不设置账号名、密码或微信绑定。
5. 账号名和密码可以独立补设；只设置密码即可使用“手机号 + 密码”。
6. 使用服务端可撤销会话支持手机号、密码和微信三种登录方式以及多设备登录。
7. 业务角色继续由服务端 `users.roles` 管理，任何登录方式均不得自动授予教练、店主或管理员权限。
8. 在事务、并发、验证码重放、会话撤销和数据库异常下保持唯一性与失败关闭。

## 非目标

首版不包含：

- 更换手机号；
- 解绑或换绑微信；
- 修改已设置的账号名；
- 多账号合并；
- 旧认证数据迁移；
- 把邮箱升级为登录凭证或重做现有邮箱找回产品流程；
- 自建 HTTP 身份服务器；
- 把管理员、支付回调或定时任务强行改成用户会话认证；
- 自动清理云端数据或删除仓库文件。

数据模型为未来换绑保留 `status`、时间戳和反向引用，但首版不暴露相关操作。

## 已确认的产品规则

### 账号与凭证

- `accountId` 是不可见、不可变的内部主体 ID。
- 账号名、手机号、密码和微信均不是账号主键。
- 账号名与账号严格 `0..1 : 1`，全平台唯一。
- 手机号与账号严格 `0..1 : 1`，全平台唯一。
- 微信与账号严格 `0..1 : 1`，全平台唯一。
- 手机号首次注册不要求账号名和密码。
- 手机号账号以后可以只设置密码，从而启用“手机号 + 密码”。
- 手机号账号以后可以再设置唯一账号名，从而同时启用“账号名 + 密码”。
- 账号名和密码注册也不自动绑定手机号或微信。

### 微信绑定

- 手机号验证码登录和手机号密码登录均不得自动绑定当前微信。
- 账号名密码注册和登录均不得自动绑定当前微信。
- 用户可在“我的 → 设置 → 账号与安全”主动绑定微信。
- 绑定前必须弹窗提示“是否绑定当前微信？绑定后，后续可直接使用微信登录”。
- 用户选择“暂不绑定”时，手机号登录仍然成功并进入账号。
- 用户选择微信登录且当前微信未绑定时，下一步进入手机号验证。
- 手机号已存在时恢复原账号；手机号不存在时创建新账号。
- 手机号验证成功后再次询问是否绑定当前微信。
- 只有用户确认且账号、微信双方均未绑定时才写入绑定；任何冲突均不得覆盖原关系。
- 已绑定微信以后可从微信入口直接恢复同一账号。

### 角色与会话

- 新建账号统一创建 `users.roles: ['member']`。
- 教练、店主继续通过现有申请和审批流程授予。
- 管理员继续使用独立管理员认证边界。
- 同一账号允许多个设备同时登录，每台设备拥有独立会话。
- 普通退出只撤销当前设备会话。
- “退出其他设备”撤销除当前设备以外的会话。
- 设置或修改密码时撤销其他设备会话，并轮换当前设备令牌。

## 规范化与命名空间

- 首版手机号只接受中国大陆 `^1\d{10}$`，服务端统一规范化为 E.164 `+86` 加 11 位号码后再计算 HMAC。前端格式校验不能替代服务端校验。
- 账号名沿用现有规则：去除首尾空白后必须为 4–20 位 ASCII，字母开头，其余仅允许字母、数字和下划线；唯一性比较统一转为小写。
- 密码不做 `trim` 或大小写转换，首版接受 6–64 个字符。
- 纯数字输入只进入手机号命名空间；账号名必须字母开头，因此“手机号或账号”的服务端分流不存在歧义。
- 确定性摘要必须包含显式命名空间和版本，例如 `phone:v1:`、`account-name:v1:`、`wechat:v1:`，防止不同凭证类型发生键空间碰撞。

## 密钥派生与轮换

- 认证 HMAC 使用独立于腾讯云短信 API 密钥的认证密钥环。环境变量至少提供 `CUETRACE_AUTH_KEY_ACTIVE_VERSION` 和对应的 `CUETRACE_AUTH_KEY_<VERSION>`；每个根密钥必须为至少 32 字节的随机值。
- 使用 HKDF-SHA256 和固定标签从版本化根密钥派生独立子密钥：`phone-binding`、`wechat-binding`、`session-token`、`sms-code`、`sms-challenge`、`auth-proof`、`rate-limit`。不同用途不得复用派生结果，也不得回退使用腾讯云 SecretKey。
- 持久手机号与微信绑定文档 ID 使用 `版本号.HMAC` 格式，记录同时保存 `keyVersion`。账号反向引用包含完整版本化 ID。
- 查询持久绑定时依次计算活动版本和配置的历史版本。命中历史版本后，只有在手机号验证码、有效微信上下文或其他对应凭证已经验证成功的事务中，才能创建活动版本文档、更新账号反向引用并停用旧文档。
- 会话、短信挑战和一次性证明的原始令牌都带非秘密密钥版本前缀；服务端据此选择验证密钥，不能遍历所有密钥造成时序差异。
- 轮换顺序为：先部署包含新旧密钥的双读代码，再切换活动版本，再迁移持久绑定，最后移除旧密钥。移除前必须确认没有旧版本持久绑定；会话旧密钥至少保留 90 天或通过 `authVersion` 主动使所有旧会话失效，短信与证明旧密钥至少保留其最大 TTL 加部署缓冲时间，限流旧密钥至少保留 24 小时并把旧窗口计数合并到活动版本。
- 密钥轮换、旧文档停用和认证版本批量失效均写脱敏审计记录；密钥值本身不得出现在日志、数据库、规格响应或错误信息中。

## 方案比较与决策

### 方案一：服务端随机会话令牌（采用）

认证成功后签发高强度随机令牌。客户端只持有原始令牌，服务端只保存令牌哈希及会话状态。用户主动调用的受保护云函数先验证会话，再通过 `accountId` 读取账号和角色。

该方案完整支持手机号与微信解耦、多设备、即时撤销、账号停用和角色实时生效。代价是必须系统性收口当前依赖 `OPENID → wechat_bindings` 的业务授权路径。

### 方案二：自包含签名令牌

把账号和有效期写入签名令牌，可减少会话查询；但账号停用、角色变更、退出其他设备和密钥轮换都需要额外版本查询，收益有限且密钥风险更高，不采用。

### 方案三：按 OPENID 保存临时账号映射

手机号登录后记录当前 `OPENID` 正在使用的账号，改动较小；但不同设备无法拥有真正独立的会话，并继续混淆“登录状态”和“微信绑定”，不采用。

## 总体架构

### 客户端

登录页只负责收集输入、协议确认、展示流程状态和安全保存当前设备会话令牌。所有用户云函数调用通过统一数据服务自动携带令牌；页面不得直接读取或写入认证集合，不得自行声明 `accountId` 或角色。

### 认证云函数

- `sendSmsCode`：校验用途与频率，生成短信验证码，写入最新挑战并调用短信供应商。
- `accountAuth`：注册账号名账号、手机号登录、密码登录、微信登录、消费一次性手机号证明、设置账号名、设置或修改密码、绑定手机号、绑定微信、查询安全状态和管理会话。
- 现有 `verifySmsCode` 的“按当前 OPENID 更新用户手机号”语义退役；验证码必须在具体认证动作中原子消费，或换成一次性短期证明。

### 会话校验

提供单一职责的服务端 `requireSession` 组件：

1. 从请求中读取原始令牌并计算哈希；
2. 读取 `auth_sessions`；
3. 校验撤销状态、空闲有效期、绝对有效期和账号认证版本；
4. 读取活动账号与 `users.roles`；
5. 返回可信 `accountId`、用户和服务端角色。

用户主动调用且涉及个人数据或业务写入的云函数必须使用该组件。微信支付回调、定时任务、管理员入口和明确的公共读取接口单独盘点并保留各自可信边界。

所有以旧 `users._id`、`_openid` 或微信绑定 ID 表示用户主体的业务外键，也必须在代码层改为内部 `accountId`。本设计不保留双主键兼容层，避免同一用户在不同登录方式下形成两套业务身份。部署前必须列出受影响集合；其中的旧测试记录只能在明确确认后清理或重建。

### OPENID 的新职责

`OPENID` 仅用于：

- 微信登录；
- 用户主动绑定微信；
- 短信发送的辅助风控维度；
- 微信平台确有要求的接口调用。

`OPENID` 不再是手机号或账号密码登录后的业务账号主键，也不得替代独立会话。

需要 `OPENID` 的微信支付等平台操作必须同时区分“业务所有者 `accountId`”和“本次平台付款人 `payerOpenid`”：独立会话授权用户操作，可信微信上下文只提供本次平台参数，不能反向成为订单所有权凭证。

## 数据模型

### `accounts`

```js
{
  _id: accountId,             // 加密安全随机、不可变
  status: 'active',
  accountNameBindingId: '',   // 可选反向引用
  phoneBindingId: '',         // 可选反向引用
  wechatBindingId: '',        // 可选反向引用
  emailBindingId: '',         // 可选恢复渠道反向引用
  passwordAlgorithm: '',      // 未设置或 'scrypt-v1'
  passwordSalt: '',           // 未设置时为空
  passwordHash: '',           // 未设置时为空
  authVersion: 1,             // 批量撤销会话
  termsAcceptedAt,
  termsVersion,
  privacyAcceptedAt,
  privacyVersion,
  createdAt,
  updatedAt
}
```

`accounts` 不保存 `_openid`，不保存明文密码，也不把角色作为第二套授权账本。
新建账号必须记录用户实际勾选的协议与隐私政策版本；没有有效同意不得完成自动建号。

### `account_names`

文档 ID 为规范化账号名的确定性摘要，用于并发唯一约束。

```js
{
  _id,
  accountId,
  account,
  accountNormalized,
  status: 'active',
  createdAt,
  updatedAt
}
```

账号与账号名的正向、反向引用必须在同一事务中检查和写入。

### `phone_bindings`

文档 ID 使用服务端密钥对规范化手机号做 HMAC，避免把明文手机号用作键。

```js
{
  _id,
  accountId,
  keyVersion,
  phoneMasked,
  status: 'active',
  verifiedAt,
  createdAt,
  updatedAt
}
```

认证集合不在 `accounts`、响应或日志中保存完整手机号。现有业务资料中的联系电话与登录手机号分属不同用途，不能被静默当作认证凭证。

### `wechat_bindings`

微信身份严格定义为当前小程序的 `(APPID, OPENID)` 组合，文档 ID 由服务端可信上下文中的两者生成确定性 HMAC。首版唯一性只承诺在当前小程序应用内成立，不通过 `UnionID` 自动合并其他应用账号。

```js
{
  _id,
  accountId,
  keyVersion,
  appidHash,
  openidHash,
  unionidHash,
  status: 'active',
  consentedAt,
  createdAt,
  updatedAt
}
```

账号反向引用与微信绑定文档必须在同一事务中写入，保证双向 `1:1`。
`UNIONID` 缺失时 `unionidHash` 为空；存在时仅作为审计元数据。原记录为空而可信上下文首次提供 `UNIONID` 时可以补写；原记录非空且后来出现不一致值时失败关闭并记录脱敏安全事件，不得覆盖、合并或迁移账号。

### `users`

```js
{
  _id: accountId,
  roles: ['member'],
  currentRole: 'member',
  role: 'member',
  nickname: '',
  avatar: '',
  createdAt,
  updatedAt
}
```

`users._id` 改为内部 `accountId`，成为业务资料和业务角色的唯一主体。认证后选择角色时仍由服务端检查 `users.roles`。

### `email_bindings`

现有邮箱绑定继续作为密码恢复渠道，不作为本轮新增登录凭证。关系改为引用内部 `accountId`，账号反向引用与邮箱绑定仍需保持唯一和事务一致。现有邮箱验证码、脱敏展示与找回安全规则继续生效；通过邮箱重置密码后必须撤销其他设备会话并轮换当前认证版本。

### `auth_sessions`

会话文档以原始令牌的 HMAC 作为不可逆查找键。

```js
{
  _id: tokenHash,
  accountId,
  keyVersion,
  authVersion,
  clientInstanceId,
  authenticatedAt,
  authenticationMethod,
  createdAt,
  lastSeenAt,
  idleExpiresAt,
  absoluteExpiresAt,
  revokedAt,
  revokeReason
}
```

原始令牌只返回一次并保存在当前客户端。服务端、数据库、日志和错误响应不得出现原始令牌。
`clientInstanceId` 只用于界面标识、短信范围和风控，不能作为“当前设备”的安全凭证。撤销时的当前设备严格指本次通过 `requireSession` 验证得到的会话文档。

### `sms_codes`

每次发送生成至少 128 位随机 `challengeId`，数据库以其 HMAC 为文档 ID。挑战不再要求手机号对应当前 `OPENID` 已有账号绑定。请求范围用于阻止一个页面或账号发起的验证码被另一个上下文消费：普通登录绑定可信 `(APPID, OPENID)` 与客户端实例，绑定手机号额外绑定当前 `accountId`，微信入口额外绑定当前微信身份。记录至少包含：

- `purpose`: `login`、`bind_phone`、`wechat_entry` 或 `reauth`；
- `keyVersion`；
- `phoneBindingId`；
- `scopeHash`；
- `generation`；
- `codeHash`；
- `createdAt`、`expiresAt`、`lastSentAt`；
- `failedAttempts`、`locked`、`used`；
- 供应商发送 claim 与结果状态。

客户端提交验证码时必须同时提交原始 `challengeId`。只有供应商状态为 `sent`、范围匹配、仍是该手机号与用途的全局最新代次、未过期、未锁定且未使用的挑战才能消费。不同用途的验证码不可交叉消费。

用途本身也决定调用权限：`login` 可在未登录状态发送；`bind_phone` 必须携带有效账号会话并绑定该 `accountId`；`wechat_entry` 必须来自当前可信微信上下文；`reauth` 必须携带有效会话，用户重新输入的完整手机号 HMAC 必须等于该账号现有 `phoneBindingId`，且挑战范围绑定当前 `accountId` 和当前会话 ID。客户端不能通过自行修改 `purpose` 绕过这些前置条件。

### `sms_rate_limits`

手机号 HMAC 和可信微信上下文分别使用确定性限流文档。手机号文档保存全用途 60 秒冷却、滚动 24 小时次数，以及每个用途的最新 `generation` 与活动 `challengeId` 哈希。新发送在事务中递增对应代次并替换活动挑战，因此同手机号、同用途的任何旧验证码都会失效，即使请求来自不同客户端实例。`clientInstanceId` 不能放宽冷却或日限额。

### `password_rate_limits`

密码失败限流使用三类确定性文档：凭证标识 HMAC 与可信微信上下文组合、可信微信上下文汇总，以及跨微信上下文的凭证标识汇总。记录窗口起点、失败次数和 `blockedUntil`，不保存明文手机号、账号名或密码。

### `auth_proofs`

微信入口手机号验证成功后签发带密钥版本前缀的 256 位随机、5 分钟有效的一次性手机号证明。服务端只保存证明 HMAC、`keyVersion`、手机号绑定键、用途、发起时的 `(APPID, OPENID)` 摘要、客户端实例、到期时间和消费状态。完成动作必须来自相同可信微信上下文。它只能被“完成微信入口”动作消费一次，不是登录会话，不能调用业务云函数。

## 核心数据流

### 共同身份完整性检查

任何登录或会话签发都必须通过统一解析器确认：凭证关系 `status === 'active'`，凭证文档 ID 与规范化输入一致，凭证的 `accountId` 与账号反向引用一致，账号 `status === 'active'`，`users._id === accountId` 且角色结构有效。微信路径还必须核对当前 `(APPID, OPENID)` 与绑定摘要；任一文档缺失、停用、非确定性或单向引用均失败关闭，不得为其签发会话。

### 手机号验证码登录

1. 用户勾选协议，输入手机号并获取 `login` 用途验证码。
2. 服务端验证格式、频率和配置，写入挑战后发送短信。
3. 用户提交验证码，服务端在事务中验证并消费最新挑战。
4. 若 `phone_bindings` 已存在，读取原账号；若不存在，在同一事务创建 `accounts`、`phone_bindings` 和默认 `users`。
5. 签发当前设备会话并返回账号与服务端角色。
6. 全流程不读取或写入 `wechat_bindings`。

新建账号、手机号关系、默认用户和首个会话必须在同一事务中写入；原始令牌在事务外预生成，只有事务成功后才返回客户端。

### 密码登录

1. 用户输入“手机号或账号”和密码。
2. 服务端根据格式选择 `phone_bindings` 或 `account_names`，解析唯一 `accountId`。
3. 使用 `scrypt` 和常量时间比较验证账号密码。
4. 账号不存在、密码错误或账号尚未设置密码统一返回 `INVALID_CREDENTIALS`；密码页固定展示“未设置密码可使用验证码登录”，不得通过错误码泄露账号或密码状态。
5. 验证成功后签发设备会话，不建立微信绑定。

会话写入前必须在同一事务中重新确认账号、凭证关系和认证版本仍有效，防止密码校验后账号被停用或凭证被撤销的竞态。

### 账号名密码注册

1. 校验协议、账号名格式、密码长度和确认密码。
2. 在事务中检查账号名未占用，创建内部账号、`account_names` 和默认 `users`。
3. 签发设备会话。
4. 不创建手机号或微信绑定。

### 微信登录入口

1. 服务端只从 `cloud.getWXContext()` 读取可信 `OPENID`。
2. 已有完整有效微信绑定时，直接恢复账号并签发设备会话。
3. 未绑定时进入手机号验证页，验证码用途为 `wechat_entry`。
4. 验证成功后生成短期一次性手机号证明。
5. 页面弹窗询问是否绑定当前微信。
6. 用户选择“暂不绑定”时，消费证明，恢复或创建手机号账号并签发会话；微信关系零写入。
7. 用户选择“确认绑定”时，消费证明，恢复或创建手机号账号，在事务中确认账号和微信双方均未绑定，建立双向微信关系并签发会话。
8. 任一侧已绑定其他对象时拒绝覆盖，不留下单边关系。

### 设置页绑定手机号

1. 验证当前设备会话。
2. 要求当前会话在最近 10 分钟内完成过密码、已绑定手机号验证码或已绑定微信验证；账号名密码账号首次绑定手机号时可重新输入当前密码。
3. 发送 `bind_phone` 用途验证码。
4. 在事务中验证并消费挑战，确认账号尚无手机号且目标手机号未被占用。
5. 写入 `phone_bindings` 和账号反向引用。
6. 绑定后立即支持手机号验证码和手机号密码登录；账号未设置密码时只支持验证码。

### 设置账号名

1. 验证当前设备会话并满足最近 10 分钟内的近期认证。
2. 服务端按固定 ASCII 规则规范化账号名，确认当前账号尚无账号名且目标确定性文档不存在。
3. 在同一事务中创建 `account_names` 并写入账号反向引用。
4. 首版设置后不可修改；并发占用返回 `ACCOUNT_NAME_EXISTS`，不得留下单向引用。

### 设置页绑定微信

1. 验证当前设备会话。
2. 要求当前会话在最近 10 分钟内完成过密码、已绑定手机号验证码或已绑定微信验证。
3. 显示明确确认弹窗。
4. 用户确认后，服务端读取当前可信 `(APPID, OPENID)`。
5. 在事务中确认当前账号与微信双方均未绑定。
6. 写入 `wechat_bindings` 和账号反向引用。

### 密码找回与修改

- 已有邮箱绑定与邮箱验证码找回流程继续使用内部 `accountId`，不得通过旧微信用户 ID 查找账号。
- 已绑定微信可继续作为受信恢复渠道，但必须通过完整有效的 `wechat_bindings` 解析账号。
- 手机号验证码登录本身已经能恢复账号，首版不另建“短信重置密码”分支；用户登录后可在“账号与安全”设置或修改密码。
- 首次设置密码和修改密码都属于敏感操作，必须具有最近 10 分钟内的手机号验证码、原密码、有效邮箱证明或已绑定微信验证；仅持有长期会话不够。
- 任何密码设置、修改或重置都必须递增账号认证版本，撤销其他设备并轮换当前会话；无当前会话的找回流程在成功后返回登录页。

### 近期重新认证

1. 敏感操作发现当前会话的 `authenticatedAt` 超过 10 分钟时返回 `RECENT_AUTH_REQUIRED` 和该账号允许使用的脱敏认证方式。
2. 手机号方式要求用户重新输入完整手机号并获取 `reauth` 用途验证码；服务端只比较 HMAC 与当前账号的既有 `phoneBindingId`，不会回传或存储完整号码。
3. 密码方式只校验当前会话对应账号的原密码，不接受客户端传入其他账号标识。
4. 微信方式要求当前可信 `(APPID, OPENID)` 与该账号现有有效微信绑定完全一致。
5. 邮箱方式消费绑定当前 `accountId` 的现有邮箱验证码证明。
6. 成功后在同一事务中消费证明或验证码、重新校验当前会话未撤销，并只更新当前会话的 `authenticatedAt` 与 `authenticationMethod`；不得签发给其他账号，也不得延长 90 天绝对期限。

## 页面与交互

### 登录页

- 顶部切换“验证码登录 / 密码登录”。
- 验证码登录包含手机号、验证码、获取验证码和“登录 / 注册”按钮。
- 明示“未注册手机号验证后将自动创建账号”。
- 密码登录的标识输入框为“手机号或账号”，保留“忘记密码”。
- 保留“注册账号”，用于创建账号名密码账号。
- 微信登录为独立入口。
- 协议与隐私政策勾选对所有登录和注册方式统一生效。
- 认证成功后进入现有服务端角色选择页。

### 微信未绑定流程

- 微信入口未解析到绑定时，直接进入手机号验证页。
- 验证成功后显示确认弹窗。
- “暂不绑定”继续手机号登录。
- “确认绑定”成功后提示“绑定成功，后续可直接使用微信登录”。

### 账号与安全

- 账号名：未设置时可设置一次；已设置时仅展示。
- 登录密码：支持首次设置和修改。
- 手机号：未绑定时进入短信绑定页；已绑定时显示脱敏号码。
- 微信账号：未绑定时弹窗确认并绑定；已绑定时显示状态。
- 登录设备：提供“退出其他设备”。

首版不提供账号名修改、手机号更换、微信解绑或换绑入口。

## 会话与安全规则

### 会话时效

- 令牌使用至少 256 位加密安全随机数。
- 连续 30 天无活动后失效。
- 无论是否活跃，单次会话最长 90 天。
- `lastSeenAt` 可按固定最小间隔节流更新，避免每次请求都写数据库，但不得延长绝对有效期。
- 账号停用、认证版本不匹配、会话撤销或过期时一律拒绝访问并要求重新登录。
- 新登录会话记录实际认证方式和 `authenticatedAt`。普通活跃请求只能更新 `lastSeenAt`，不能延长“近期认证”时间。

### 近期认证

- 绑定手机号、绑定微信、首次设置或修改密码、设置账号名和退出其他设备均要求最近 10 分钟内完成过有效凭证验证。
- 可接受的凭证是当前账号原密码、已绑定手机号验证码、有效邮箱证明或已绑定微信验证；具体操作只能使用不会循环依赖目标新凭证的方式。
- 微信入口手机号证明本身属于刚完成的手机号验证，可用于紧接着的微信绑定确认。
- 仅持有最长 90 天的会话令牌不能升级、替换或新增永久登录凭证。

### 会话撤销

- 退出当前设备：撤销当前令牌。
- 退出其他设备：完成近期认证后递增账号 `authVersion`，严格排除本次已验证会话，并为该会话签发新版本令牌。
- 设置或修改密码：递增 `authVersion`，撤销其他设备并轮换当前设备令牌。
- 同一客户端重复登录产生的旧会话也属于“其他会话”，会在退出其他设备或密码变更时一并失效。
- 客户端收到会话无效结果时清除本地认证状态并返回登录页。

### 短信验证码

- 6 位加密安全随机数字；
- 5 分钟有效；
- 同一手机号所有用途共享 60 秒发送冷却；
- 同一手机号滚动 24 小时最多接受 10 次供应商发送尝试；
- 同一可信 `(APPID, OPENID)` 滚动 24 小时最多接受 30 次发送尝试；
- 最多错误 5 次，第五次锁定并失效；
- 新验证码覆盖同手机号、同用途的旧验证码；
- 成功验证或生成证明后一次性消费；
- 数据库只保存验证码 HMAC；
- 只有供应商确认 `sent` 的当前代次挑战可以消费，`pending`、`failed`、`superseded` 均不可消费；
- 所有冷却、日限额、代次替换和发送 claim 必须原子更新，客户端实例不能绕过全局限制。

首次发送时验证码文档不存在属于正常路径。可选文档读取只能捕获经过验证的“文档不存在”错误；权限、网络、事务和其他数据库错误不得伪装成空文档。

### 密码

- `scrypt-v1` 固定参数为 `N=16384`、`r=8`、`p=1`、输出 64 字节、随机盐 16 字节、`maxmem=64 MiB`，并使用常量时间比较；实现不得依赖运行时默认参数；
- 首版按已确认的产品兼容要求使用 6–64 个字符规则；安全性主要由慢哈希、近期认证和多维限流补强；
- 明文密码不得落库、返回或写日志；
- 同一凭证标识与可信微信上下文组合在 15 分钟内最多失败 5 次，超限后该组合暂停 15 分钟；
- 同一可信微信上下文在 15 分钟内最多提交 20 次失败密码验证；
- 同一凭证标识跨全部微信上下文在滚动 24 小时内最多失败 30 次，超限后该标识暂停密码登录 60 分钟；手机号验证码登录和已绑定恢复渠道仍可用；
- 账号不存在、账号未设置密码、密码哈希结构无效和密码错误均必须执行同参数、同输出长度的虚拟 `scrypt` 后再返回统一错误，降低时序枚举；
- 不设置可被任意攻击者触发的永久账号锁，避免利用登录接口对目标账号实施拒绝服务。

### 数据权限与隐私

- `accounts`、`account_names`、`phone_bindings`、`wechat_bindings`、`email_bindings`、`email_codes`、`auth_sessions`、`sms_codes`、`sms_rate_limits`、`password_rate_limits`、`auth_proofs`、`auth_control` 和 `users` 均禁止小程序客户端直接读写。
- 客户端只接收业务必要的账号展示名、脱敏手机号、绑定状态、角色和原始会话令牌。
- 日志不得打印手机号、验证码、密码、原始会话令牌或密钥。
- 隐私政策需披露登录会话、验证手机号、短信发送记录和当前微信绑定的用途、保存范围与用户权利。

## 错误处理

| 错误码 | 含义 | 客户端处理 |
|---|---|---|
| `INVALID_PHONE` | 手机号格式错误 | 提示修正输入 |
| `SMS_TOO_FREQUENT` | 验证码发送过频 | 保留输入并提示稍后重试 |
| `SMS_CODE_INVALID` | 验证码错误 | 允许在剩余次数内重试 |
| `SMS_CODE_EXPIRED` | 验证码过期 | 引导重新获取 |
| `SMS_CODE_LOCKED` | 错误次数达到上限 | 要求重新获取验证码 |
| `INVALID_CREDENTIALS` | 账号不存在、未设置密码、密码结构无效或密码错误 | 统一执行 scrypt 并使用统一文案，避免账号枚举 |
| `PASSWORD_RATE_LIMITED` | 密码失败次数达到临时阈值 | 提示稍后重试或改用验证码/已绑定恢复渠道 |
| `ACCOUNT_NAME_EXISTS` | 账号名已被占用 | 提示更换账号名 |
| `PHONE_ALREADY_BOUND` | 手机号属于其他账号 | 完成手机号验证后提示使用原账号或联系客服 |
| `ACCOUNT_PHONE_ALREADY_BOUND` | 当前账号已有手机号 | 首版拒绝覆盖 |
| `WECHAT_ALREADY_BOUND` | 当前微信属于其他账号 | 拒绝覆盖，提示换绑功能暂未开放 |
| `ACCOUNT_WECHAT_ALREADY_BOUND` | 当前账号已有微信 | 首版拒绝覆盖 |
| `WECHAT_IDENTITY_CONFLICT` | APPID、OPENID 或 UnionID 审计关系异常 | 失败关闭并联系支持，不自动合并 |
| `SESSION_REQUIRED` | 缺少或无效会话 | 清理本地状态并返回登录页 |
| `SESSION_EXPIRED` | 会话过期 | 清理本地状态并重新认证 |
| `RECENT_AUTH_REQUIRED` | 敏感操作需要近期凭证验证 | 展示当前账号可用的密码、手机号、邮箱或微信验证入口 |
| `CLIENT_UPDATE_REQUIRED` | 客户端认证协议低于服务端下限 | 阻止业务操作并引导更新小程序 |
| `AUTH_MAINTENANCE` | 认证体系处于协调发布维护状态 | 停留在维护提示页，禁止重试写入 |
| `ACCOUNT_DISABLED` | 账号已停用 | 拒绝登录和业务访问 |
| `ROLE_NOT_ALLOWED` | 所选角色未获授权 | 刷新角色并返回选择页 |
| `AUTH_CONFLICT` | 并发导致状态变化 | 刷新状态，不自动覆盖 |
| `AUTH_INTERNAL_ERROR` | 数据库或服务异常 | 失败关闭，不进入 mock 或业务页 |

所有创建账号、创建凭证关系、消费验证码或证明、绑定关系和会话版本变更均应使用事务或等价原子约束，异常时不得留下半个账号或单边绑定。
当认证动作同时改变账号或绑定并签发会话时，会话文档必须包含在同一事务中；如果响应在事务成功后丢失，未送达客户端的孤立会话只能等待撤销或过期，不能重放原始令牌。

## 实施分段与发布约束

当前静态扫描发现 93 个云函数入口文件引用 `getWXContext()`、`wechat_bindings` 或 `_openid`。这是待审计上限，不代表 93 个入口都应改成用户会话；它说明本次不是单一登录页修改。实施计划必须先按“用户受保护入口、公共读取、管理员、支付回调、定时任务、平台专用 OPENID”分类，再按业务域分批迁移并运行 focused 测试。

实施开始时必须生成 `docs/auth-v2-migration-matrix.md`，逐项列出：云函数或集合、当前身份键、新身份键、认证类别、读写外键、focused 测试、发布批次和完成状态。矩阵未覆盖的入口不得进入发布清单。

开发可分为以下内部阶段，但不得把不兼容阶段分别发布到生产：

1. 建立新账号、凭证、证明和会话核心，并通过云函数测试。
2. 将用户主动调用的受保护云函数按个人资料/社交、训练与预约、教练与门店、球桌订单与支付等业务域迁移到 `requireSession`，并逐个审计公共、管理员、支付回调和定时任务边界。
3. 改造登录页、账号与安全页、统一数据服务和本地会话处理。
4. 完成全量回归、部署文档、权限配置和真机验收。

发布必须使用维护窗口协调更新认证云函数、受保护业务云函数和小程序客户端，避免新旧身份模型混用。

### 版本闸门与发布顺序

1. 新增服务端只读的 `auth_control` 配置，至少包含 `maintenance`、`schemaVersion` 和 `minClientProtocol`。
2. 先发布一轮向后兼容的协议守卫到迁移矩阵内所有用户入口；此时仍允许协议 v1，但已能统一进入维护状态和拒绝过旧客户端。
3. 验证所有入口都受守卫控制后，将 `maintenance` 设为 `true`，阻止用户读写，不依赖客户端页面自行停用。
4. 在维护状态下创建 v2 集合与索引，部署认证核心、会话校验、所有业务域迁移和新客户端；新客户端固定提交 `authProtocol: 2`。
5. 在维护状态下运行云端冒烟测试并核对迁移矩阵，确认没有残留 v1 用户入口。
6. 经明确确认后清理或重建旧测试数据，将 `schemaVersion` 和 `minClientProtocol` 切为 2，再解除维护。
7. 缓存的旧客户端或漏传协议的请求统一收到 `CLIENT_UPDATE_REQUIRED`，不得继续写旧身份数据。

### 回滚

- 解除维护前失败：保持维护，直接回滚尚未开放的云函数包和配置，不需要处理用户流量数据。
- 解除维护后失败：先恢复维护并保留现场，优先前向修复。若必须退回 v1，先列出 v2 期间产生的账号和业务测试记录；任何清理仍需再次确认，再回滚函数包并把协议下限恢复为 1。
- 不允许部分业务域退回 v1、其他业务域继续 v2，也不允许用双主键静默兼容掩盖失败。

部署前需要：

- 创建集合与必要索引；
- 将认证集合权限设置为小程序端不可读写；
- 配置版本化认证密钥环，并验证手机号、微信、验证码、挑战、证明、会话与限流使用各自 HKDF 派生子密钥；腾讯云短信 API 密钥不得兼作认证 HMAC；
- 更新部署说明和隐私政策；
- 明确列出准备清理的认证集合，以及所有引用旧用户 ID 的业务测试集合与影响范围，再由用户确认执行；
- 新客户端首次启动时清除旧本地认证缓存，但保留无关业务设置。

必要索引至少包括：`auth_sessions(accountId, authVersion, revokedAt)`、`auth_sessions(idleExpiresAt)`、`auth_sessions(absoluteExpiresAt)`、`sms_codes(expiresAt, status)`、`auth_proofs(expiresAt, used)`。手机号、账号名和微信唯一性由带版本命名空间的确定性文档 ID、账号反向引用和事务共同保证。过期短信、证明和会话由定时清理任务处理；清理任务不得改变其在到期瞬间已经失效的授权语义。

## 测试策略

### 认证核心

1. 新手机号验证码登录原子创建 `accounts`、`phone_bindings` 和默认 `users`。
2. 已有手机号恢复同一账号，不重复创建。
3. 手机号、账号名和微信的双向 `1:1` 在并发下成立。
4. 手机号账号设置密码后支持手机号密码登录；设置账号名后同时支持账号名密码登录。
5. 账号名密码注册不创建手机号或微信绑定。
6. 密码、验证码、证明和会话令牌不以明文落库或返回到非必要响应。
7. 现有邮箱绑定与邮箱找回改用内部 `accountId`，密码重置后旧会话全部失效。
8. 未同意有效协议版本不能自动建号；成功建号保存协议和隐私政策版本与时间。
9. 手机号 E.164、账号名大小写和命名空间规范化生成稳定且互不冲突的凭证键。
10. 停用账号、失效凭证、单向引用或异常微信身份均不得签发会话。
11. 活动与历史 HMAC 密钥可双读，成功验证后原子迁移持久绑定；过早移除历史密钥会被部署检查阻止。

### 微信流程

1. 已绑定微信直接恢复原账号。
2. 未绑定微信配合新手机号创建新账号。
3. 未绑定微信配合已有手机号恢复原账号。
4. “暂不绑定”零微信写入但登录成功。
5. “确认绑定”建立完整双向关系。
6. 账号侧或微信侧冲突时零覆盖、零单边写入。
7. 手机号证明过期、复用或用途错误均失败。
8. 微信身份严格按 `(APPID, OPENID)` 区分，缺失 UnionID 可用，冲突 UnionID 失败关闭且不合并账号。

### 会话与授权

1. 每台设备令牌独立，多设备会话并存。
2. 当前设备退出只撤销当前令牌。
3. 退出其他设备和密码变更使旧版本令牌失效，并保留轮换后的当前会话。
4. 30 天空闲和 90 天绝对期限均生效。
5. 账号停用、角色撤销在下一次受保护请求立即生效。
6. 所有用户受保护入口拒绝缺失、伪造、过期或撤销会话。
7. 管理员、支付回调、定时任务和公共接口保持独立可信边界。
8. 所有用户业务外键使用内部 `accountId`，同一账号通过不同凭证登录后读取到同一份资料和业务记录。
9. 微信支付订单使用独立会话确认业务所有者，`payerOpenid` 只作为本次平台付款参数，不能越权访问其他账号订单。
10. 被盗但未经过近期认证的长期会话不能绑定新手机号/微信、设置密码、设置账号名或退出其他设备。
11. “当前设备”按当前会话令牌识别；伪造 `clientInstanceId` 不能保留或撤销其他会话。
12. 手机号、原密码、邮箱和微信重新认证只更新当前有效会话，绑定其他账号或复用证明均失败。

### 短信与数据库语义

1. 首次 `sms_codes` 文档不存在时创建成功并调用短信供应商。
2. 测试桩模拟真实 CloudBase 缺失 `doc.get()` 抛错语义。
3. 60 秒冷却、5 分钟过期、五次锁定、新码覆盖旧码和一次性消费全部覆盖。
4. 登录、绑定和微信入口验证码不可跨用途消费。
5. 短信供应商失败、数据库权限错误、事务冲突和超时均失败关闭。
6. 同手机号、同用途的新代次使不同客户端范围的旧挑战失效；客户端实例不能绕过 60 秒或 24 小时限额。
7. 只有供应商状态为 `sent` 的活动代次可消费，`pending`、`failed` 和 `superseded` 均失败。
8. 密码失败次数和 15 分钟暂停阈值按凭证与可信上下文正确生效，且不会形成永久账号锁。
9. 未知账号、未设密码、损坏哈希和错误密码都执行相同参数的虚拟 `scrypt` 并返回统一错误。
10. 跨多个微信上下文的同一凭证聚合限流在 30 次失败后暂停 60 分钟，验证码恢复渠道仍可用。

### 页面行为

1. 验证码登录与密码登录布局、按钮和提示符合设计。
2. 重复点击只发起一次发送、登录或绑定请求。
3. 页面切换、返回或卸载后，迟到请求不得更新状态或弹出错误提示。
4. 微信确认弹窗的确认和取消分支均正确。
5. 会话失效统一清理状态并回到登录页。
6. 身份选择只展示服务端角色并由服务端再次校验。
7. 维护状态阻止所有用户写入；协议 v1 或缺失协议的旧客户端在切换后收到 `CLIENT_UPDATE_REQUIRED`。

### 最终验证

- 行为修改先运行 focused 测试；
- 最终由根任务运行一次 `scripts/codex-verify.ps1`；
- 对改动的 JavaScript 执行语法检查；
- 检查 diff、文本编码、敏感信息和旧 `OPENID` 授权路径；
- 在真实云环境和真机上完成短信与微信专项验收。

## 验收标准

1. 新手机号输入正确验证码后直接创建并进入同一内部账号，无账号名、密码或微信绑定。
2. 手机号账号设置密码后，可使用该手机号与同一密码登录。
3. 手机号账号设置账号名后，可同时使用账号名与同一密码登录。
4. 账号名密码账号绑定手机号后，可使用账号名密码、手机号密码或手机号验证码登录同一账号。
5. 手机号、账号名和微信任何一项都不能绑定多个账号，也不能被并发覆盖。
6. 手机号或账号密码登录不会自动建立微信绑定。
7. 微信入口未绑定时必须先验证手机号并展示确认弹窗；取消仍可登录，确认后下次可直接微信登录。
8. 所有登录方式创建的新账号只有 `member`，不能通过客户端输入获得其他角色。
9. 多设备会话、当前退出、退出其他设备、密码变更撤销和 30/90 天过期符合设计。
10. 受保护业务云函数不再依赖 `OPENID` 直接推断手机号或账号密码登录后的账号。
11. 首次获取短信验证码不再因缺失 `sms_codes` 文档失败，真机能够收到并消费验证码。
12. 数据库中不存在明文密码、验证码或原始会话令牌，客户端不能直接读写认证集合。
13. 现有邮箱绑定与密码找回继续可用，并在密码重置后正确撤销旧会话。
14. 同一账号通过手机号、账号名或微信登录后读取到相同的用户资料、角色和业务记录。
15. 长期会话未经近期凭证验证不能新增或修改永久凭证；手机号单凭证账号可通过 `reauth` 验证码完成敏感操作。
16. 认证密钥轮换期间新旧绑定均可解析，历史密钥不能在持久绑定迁移完成前移除。
17. 自动化测试、最终验证脚本和真机专项验收均有明确通过结果。

## 决策记录

| 决策 | 选择 | 理由 |
|---|---|---|
| 账号主体 | 不可变内部 `accountId` | 手机号、账号名和微信变化不改变业务主体 |
| 手机号唯一性 | 手机号与账号严格 `1:1` | 避免手机号密码登录歧义和静默抢绑 |
| 微信绑定时机 | 仅用户主动确认 | 登录与绑定解耦，尊重用户授权 |
| 手机号首次登录 | 自动建号、不设账号名密码 | 满足主流验证码即注册体验 |
| 密码与账号名 | 独立设置 | 只设密码即可启用手机号密码登录 |
| 登录会话 | 服务端随机可撤销令牌 | 支持多设备、即时撤销和角色实时生效 |
| 登录页布局 | 验证码 / 密码双栏 | 路径短，手机号与账号名密码入口兼容 |
| 默认角色 | `member` | 登录方式不应授予业务权限 |
| 首版绑定范围 | 仅首次绑定 | 先稳定认证底座，换绑与申诉另立规格 |
| 数据处理 | 测试数据清理重建 | 当前无须保留的生产账号，避免复杂迁移 |
| 会话期限 | 空闲 30 天、绝对 90 天 | 兼顾主流体验与长期令牌风险 |
| 敏感操作 | 10 分钟内近期认证 | 防止长期会话被盗后升级为永久接管 |
| 密码兼容 | 6–64 字符 + 固定 scrypt + 多维限流 | 保留已确认的六位密码场景并补强在线与离线防护 |
| HMAC 生命周期 | 版本化密钥环 + HKDF 子密钥 + 双读迁移 | 避免密钥轮换使持久手机号或微信绑定失联 |
