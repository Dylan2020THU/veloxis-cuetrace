## 二、错误码详细排查

### 268452661 — 风险共建 API 参数错误

排查步骤：

1. 看响应体 `message` 与 `detail.field` 定位具体错误字段

2. `notify_url` 格式严格校验：

   - 公网域名必须 `https://`（专线接入可 `http://`）

   - 不能携带 query 参数（如 `?token=xxx`）或 fragment（如 `#section`）

   - 长度 ≤ 256 字符

   - 不能含非法字符 / 中文 / 空格
3. `notify_url` 必须外网可达（DNS、端口、HTTPS 证书都需正常）

4. 字段名是下划线风格 `notify_url`，不要写成 `notifyUrl`

5. GET / DELETE 接口不需要传 body

6. 配置前可用 curl 模拟微信支付 POST 请求验证 URL 可达性


* * *

### 270924345 — content-type 不正确

排查步骤：

1. 打印实际请求头 `Content-Type`，POST / PUT 必须严格等于 `application/json`

2. GET / DELETE 可以直接不带 `Content-Type`，但如果带了必须是 `application/json`

3. 部分 HTTP 客户端 GET / DELETE 默认加 `text/plain` 需要清除或显式覆盖

4. HTTP 客户端封装时在 baseClient 层固定设置

5. 抓包确认实际发送的 Header（部分 SDK 默认 Header 与代码声明不一致）


* * *

### 269546505 — 商户证书已过期

排查步骤：

1. 登录服务商商户平台 → 账户中心 → API 安全 → API 证书，查看到期时间

2. 重新申请并下载新证书，更新到所有服务节点

3. 建议在证书过期前 30 天监控告警

4. 长期方案可迁移到微信支付公钥体系


* * *

### 268452662 — 风险共建 API 查询失败

排查步骤：

1. 后端依赖服务瞬时异常，可使用相同参数稍后重试（5s → 30s → 1min 退避）

2. 重试前确认是否命中频率限制——业务侧不要高频轮询查询接口

3. 切换备域名重试（ `api2.mch.weixin.qq.com`）

4. 业务侧把回调地址配置本地缓存，不需要每次都查 API

5. 长时间持续失败时联系微信支付技术支持


* * *

### 269546525 — 错误的签名，导致验签失败

排查步骤：

1. 完整打印「签名前的待签名串」与「实际请求 body」

2. 签名串拼接： `HTTP方法\nURL路径(含 query)\n时间戳\n随机串\n请求体\n`

3. 4 个 HTTP 方法（GET / POST / PUT / DELETE）的签名串首行必须严格一致

4. GET / DELETE 请求 body 用空字符串（末尾仍要 `\n`）

5. 核对 `apiclient_key.pem` 是当前 SP 商户号对应的私钥

6. 中间链路（如 Nginx / API 网关）禁止改写 body

7. 建议使用官方 SDK


* * *

### 269546507 — 商户证书序列号有误

排查步骤：

1. 用 OpenSSL 提取证书序列号确认： `openssl x509 -in apiclient_cert.pem -noout -serial`

2. 把输出中 `serial=` 后面的部分（去掉冒号、不要 `0x` 前缀）填到 `Authorization.serial_no`

3. `serial_no` 是商户 API 证书序列号，不是平台证书序列号

4. 推荐使用官方 SDK 自动管理证书序列号


* * *

### 269546538 — 商户号不存在

排查步骤：

1. 仔细核对服务商商户号（10 位纯数字），前后不能有空格

2. 业务侧入参做正则校验： `^[0-9]{10}$`

3. 检查请求头 `Authorization` 中签名使用的 `mchid` 与代码中传递的 `mchid` 是否一致


* * *

### 268460485 — 商户违规回调地址不存在

排查步骤：

1. 首次接入需先调 `POST /v3/merchant-risk-manage/violation-notifications` 创建回调地址

2. 该错误是正常业务态，业务侧应当作「未配置」处理，不要重试 / 不要告警

3. 推荐启动时一次性逻辑：先 GET 查询，若 404 则 POST 创建；若存在但 URL 不一致则 PUT 更新

4. 维护「服务商号 → 是否已创建回调」状态，避免每次启动都误调


* * *

### 268460486 — 商户违规回调地址已存在

排查步骤：

1. 业务侧应做幂等性处理——把该错误码当作「已成功」

2. 每个服务商号仅能配置一个回调地址，要更换地址只能用 PUT 修改

3. 流程优化：先 GET 查询是否已存在，再决定 POST / PUT：

   - 不存在则 POST 创建

   - 存在但 URL 不一致则 PUT 修改

   - 存在且 URL 一致则跳过
4. 多实例场景加分布式锁，单服务商号单回调配置串行化

5. 监控告警规则中过滤该错误码
