## 二、错误码详细排查

### 269546505 — 商户证书已过期

排查步骤：

1. 登录服务商商户平台 → 账户中心 → API 安全 → API 证书，查看到期时间

2. 重新申请并下载新证书，更新到所有服务节点

3. 服务商对接多个子商户时证书过期影响面大，建议在证书过期前 30 天监控告警

4. 长期方案可迁移到微信支付公钥体系，规避证书定期轮换


* * *

### 269761029 — 提交时申请单状态未知

排查步骤：

1. 稍后重试：使用相同的 `out_request_no` 重发请求（接口支持幂等）

2. 重试策略：5s → 30s → 1min 指数退避

3. 重试前先调「通过业务申请编号查询申请状态」确认实际状态

4. 长时间持续失败可切换备域名（ `api2.mch.weixin.qq.com`）重试


* * *

### 270924346 — 签名信息非法

排查步骤：

1. `Authorization` 头必须以 `WECHATPAY2-SHA256-RSA2048 ` 开头（注意空格）

2. 必须包含五个字段： `mchid` / `nonce_str` / `signature` / `timestamp` / `serial_no`

3. 字段间用英文逗号分隔，字段值用英文双引号包裹

4. `signature` 必须是标准 Base64 编码（非 URL-Safe）

5. `serial_no` 是当前商户 API 证书的序列号（不是平台证书）

6. 建议使用官方 SDK 自动生成 Authorization


* * *

### 269546538 — 商户号不存在

排查步骤：

1. 仔细核对服务商商户号（10 位纯数字），前后不能有空格

2. 区分服务商商户号（签名所用）与 `sub_mchid`（请求中的子商户号）

3. 业务侧入参做正则校验： `^[0-9]{10}$`

4. 检查请求头 `Authorization` 中签名使用的 `mchid` 与代码中传递的 `mchid` 是否一致


* * *

### 268552178 — 已经提交过代理签约

排查步骤：

1. 调「通过业务申请编号查询申请状态」确认当前 `applyment_state`

2. `NEED_SIGN`（待签约）/ `FINISH`（已完成）状态下不需要再提交申请

3. 驳回状态（ `REJECTED`）可用相同 `out_request_no` 覆盖修改

4. 其他状态需用新的 `out_request_no` 提交新申请单

5. 业务侧应做幂等性处理——重复提交时把这个错误码当作「已成功」


* * *

### 269761032 — 不属于电商平台商户，暂无权限

排查步骤：

1. 登录商户平台 → 产品中心 → 我的产品，确认「平台收付通-电商交易解决方案」已开通

2. 普通服务商不等于电商平台服务商，需要单独申请开通

3. 不同子能力可能需要单独申请（如二级商户进件、分账等）

4. 检查商户号是否被风控冻结导致权限被收回


* * *

### 270924291 — 头部信息不完整

排查步骤：

1. 打印实际发送的所有请求头检查

2. 必填请求头： `Authorization` / `Accept` / `User-Agent`，POST 还需 `Content-Type`

3. 含敏感字段的接口必填 `Wechatpay-Serial` 头

4. 多层代理 / 网关可能删除部分头，做链路抓包定位


* * *

### 269761031 — 参数错误，请检查后重试

排查步骤：

1. 看响应体 `message` 与 `detail.field` 定位具体错误字段

2. 重点检查：

   - `out_request_no` ≤ 124 字符，字符集 `[a-zA-Z0-9_\-*]`

   - `organization_type` 必须是 `2401` / `2500` / `4` / `2` / `3` / `2502` / `1708`

   - `business_license_info.business_license_number`：个体户 15 位数字 或 18 位以 9 开头的统一社会信用代码；企业 18 位以 9 开头的统一社会信用代码

   - 身份证号：17 位数字 + 1 位数字 / X

   - 联系手机号：11 位中国大陆手机号

   - 银行账号：纯数字，长度 5-32
3. 不同主体类型必填字段不同，参照官方文档对照

4. 重试前先修复参数，不要使用相同参数重试


* * *

### 268435461 — 参数非法

排查步骤：

1. 看响应体 `message` 与 `detail.field` 定位

2. 检查必填字段是否齐全： `business_code`、 `organization_type` 等

3. 检查敏感字段是否加密：身份证号、姓名等必须用微信支付公钥加密

4. 加密算法： `RSA/ECB/OAEPWithSHA-1AndMGF1Padding`

5. `Wechatpay-Serial` 头必须与加密所用公钥匹配

6. 图片 `MediaID` 必须是文件上传接口返回的 ID

7. 建议使用官方 SDK 自动处理敏感字段加密


* * *

### 269761028 — 存在流程进行中的申请单，请检查是否重入

排查步骤：

1. 首要：调「通过业务申请编号查询」确认当前 `applyment_state`

2. 状态对应操作：

   - `REJECTED`（已驳回）：可用相同 `out_request_no` 覆盖修改重提

   - `CHECKING` / `AUDITING` / `ACCOUNT_NEED_VERIFY` / `NEED_SIGN`：等待状态流转，不要重复提交

   - `FINISH` / `FROZEN` / `CANCELED`：用新的 `out_request_no` 创建新申请单
3. 把这个错误码当作软失败——首次请求大概率已成功

4. 业务侧重试前一定先查状态


* * *

### 269761035 — 当前申请单的父商户与服务商号不匹配

排查步骤：

1. 核对调用方服务商商户号——必须与首次提交申请单时使用的服务商号完全一致

2. 查询接口也要用同一商户号调用

3. 业务侧把「out\_request\_no → 创建方服务商商户号」绑定落库，操作前先校验归属

4. 多服务商商户号场景下严格按「一个 sp\_mchid 一套 out\_request\_no 命名空间」管理


* * *

### 269611025 — 未注册财付通

排查步骤：

1. 该错误多出现在小微商户进件场景：经营者身份证未在财付通注册过

2. 解决方案：让经营者先用本人微信完成实名认证（绑定身份证 \+ 银行卡 \+ 手机号），等待几小时同步后重新提交申请

3. 检查 `id_card_info.id_card_number` 是否填写正确

4. 个人主体（小微 / 个人卖家）特别容易遇到此问题


* * *

### 269546525 — 错误的签名，导致验签失败

排查步骤：

1. 签名串拼接： `HTTP方法\nURL路径\n时间戳\n随机串\n请求体\n`

2. URL 用绝对路径（包含 query string，不含域名）

3. POST 请求的报文主体必须是实际发送的 JSON 字符串（不要美化）

4. 签名算法 SHA256withRSA，使用商户 API 证书私钥

5. `signature` 必须是标准 Base64 编码（非 URL-Safe）

6. 建议使用官方 SDK 自动签名


* * *

### 270924332 — 认证信息解析失败

排查步骤：

1. `Authorization` 头必须以 `WECHATPAY2-SHA256-RSA2048 ` 开头（注意空格）

2. 各字段用英文逗号分隔，字段值用英文双引号包裹

3. `timestamp` 为秒级 Unix 时间戳（10 位）

4. `nonce_str` 为 32 位随机字符串

5. 建议使用官方 SDK 自动生成 Authorization 头


* * *

### 269546542 — 平台私钥解密失败

排查步骤：

1. 敏感字段加密的公钥与请求头 `Wechatpay-Serial` 必须一致

2. 平台证书会定期轮换（约 12 个月），使用旧证书加密的请求会失败：

   - 调「获取平台证书」接口拉取最新证书

   - 缓存多份证书，覆盖轮换过渡期
3. 推荐迁移到微信支付公钥体系，长期有效无需轮换

4. 加密算法： `RSA/ECB/OAEPWithSHA-1AndMGF1Padding`

5. 建议使用官方 SDK 自动管理平台证书 / 公钥


* * *

### 269761047 — 你进件特约商户的权限已被受限

排查步骤：

1. 登录服务商商户平台 → 消息 / 违约记录，查看具体管控类型与解除条件

2. 按指引申诉处理

3. 申诉解除后约几小时到几天生效

4. 期间业务侧应暂停新进件，避免堆积


* * *

### 270924344 — 使用了不支持的 Accept-Language 值

排查步骤：

1. 多数情况下不需要设置 `Accept-Language` 头，不传则使用默认值 `zh-CN`

2. 微信支付目前支持： `zh-CN`（简体中文）/ `en-US`（英文，部分接口）

3. HTTP 客户端可能默认设置了复合值（如 `zh,en;q=0.9,...`），需要清除或显式覆盖


* * *

### 269546507 — 商户证书序列号有误

排查步骤：

1. `serial_no` 必须是当前正在使用的商户 API 证书的序列号（不是平台证书）

2. 商户平台 → 账户中心 → API 安全 → API 证书，对比序列号

3. 证书重新下载后旧的 `serial_no` 失效

4. 多实例部署确认所有节点使用同一份证书


* * *

### 268524178 — 校验实名不一致

排查步骤：

1. 仔细核对以下字段在同一申请单内必须严格一致：

   - `id_card_info.id_card_name`（身份证姓名）

   - `id_card_info.id_card_number`（身份证号）

   - `business_license_info.legal_person`（营业执照法定代表人）

   - `account_info.account_name`（结算账户户名）
2. 包括姓名前后无空格、繁简体一致、特殊字符一致

3. 检查敏感字段是否加密正确（加密错误会被解密为乱码导致校验失败）

4. 检查营业执照与身份证是否最新、清晰可见


* * *

### 268552247 — 申请单状态不匹配

排查步骤：

1. 调「通过业务申请编号查询」确认当前 `applyment_state`

2. 不同接口的状态前置条件：

   - 修改 / 查询结算账户：要求 `applyment_state=FINISH`

   - 签约相关：要求 `applyment_state=NEED_SIGN` 或 `sign_state=UNSIGNED`
3. 业务侧应严格按状态机操作
