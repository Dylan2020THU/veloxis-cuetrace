## 二、错误码详细排查

### 268484120 — 授权状态不是已授权

排查步骤：

1. 用户必须先在商圈小程序通过「商圈会员快速积分插件」完成授权

2. 调「商圈会员积分服务授权查询」接口确认用户授权状态： `OPENED` / `CLOSED` / `UNAUTHORIZED`

3. 商户侧应监听「商圈会员积分服务授权结果通知」持久化每个用户的授权状态

4. 引导未授权 / 已关闭授权的用户重新到商圈小程序授权


* * *

### 268483831 — AppID 与 OpenID 不匹配

排查步骤：

1. `openid` 是 AppID 维度的，不同 AppID 下不同

2. 商圈接口中的 `appid` 必须是用户授权积分时使用的商圈小程序或服务号 AppID

3. 检查 `appid` 与 `openid` 是否成对（小程序 AppID 拿到的 openid 不能用于公众号 AppID）


* * *

### 272955914 — 订单未提交积分

排查步骤：

1. 商圈会员积分流程需先有用户在快速积分插件中提交积分申请，商户才能调积分同步接口

2. 业务侧应监听插件的积分申请通知，匹配到对应的 `transaction_id` 后才发起积分同步

3. 不要在收到支付结果通知后立即同步积分，应等待用户主动提交申请


* * *

### 268483371 — 请求的 appid 有误

排查步骤：

1. `appid` 长度 ≤ 128 字符，格式为 `wx` \+ 16 位字符或 18 位字符

2. 检查 `appid` 是否被截断 / 编码污染

3. 必须是已认证的小程序 / 公众号 / 移动应用 AppID

4. 测试号 / 沙箱 AppID 不能用于生产环境


* * *

### 268483844 — brandid 与 appid 不匹配

排查步骤：

1. 登录商户平台 → 产品中心 → 智慧商圈 → 配置管理 → 添加下单 AppID

2. 商圈小程序、商圈服务号需要分别绑定各自的 AppID

3. 一个 brandid 可绑定多个 AppID，但每个 AppID 必须显式添加

4. 绑定后约几分钟生效


* * *

### 269546505 — 商户证书已过期

排查步骤：

1. 登录商户平台 → 账户中心 → API 安全 → API 证书，查看到期时间

2. 重新下载新证书，更新到所有服务节点

3. 建议在证书过期前 30 天监控告警


* * *

### 272955915 — 订单积分已回传

排查步骤：

1. 商圈会员积分同步接口不支持重复调用——一笔交易只能回传一次积分

2. 业务侧应做幂等性控制：同一个 `transaction_id` 只处理一次

3. 因网络原因不确定是否成功时优先通过查询验证而不是盲目重发

4. 如果是异常重试导致的重复回传，把该错误码当作「已完成」处理


* * *

### 268483828 — mchid 与 brandid 关系不匹配

排查步骤：

1. 智慧商圈接口的调用方必须是该 `brandid` 的品牌主商户号或品牌服务商商户号

2. 商户平台 → 产品中心 → 智慧商圈，确认当前商户号在 brandid 下的角色

3. 检查 `brandid` 是否填错（混淆了不同品牌的 brandid）


* * *

### 268483829 — OpenID（无效或异常）

排查步骤：

1. `openid` 长度 ≤ 128 字符，字符集为大小写字母、数字、下划线 `_`、连字符 `-`

2. 检查 `openid` 是否被截断 / 编码污染

3. 不能为空字符串

4. 商圈接口要用 `openid` 而非 `unionid`


* * *

### 268435461 — 参数非法

排查步骤：

1. 看响应体 `detail.field` 定位具体非法字段

2. 时间字段 `points_update_time` 必须为 RFC3339 格式 `yyyy-MM-DDTHH:mm:ss+TIMEZONE`

3. 枚举值需在文档定义范围内（如 `points_commit_status` 必须是 `PENDING` / `FINISHED`）

4. `earn_points=false` 时必须传 `no_points_remarks`

5. `earn_points=true` 时 `increased_points` 应 \> 0

6. 必填字段不能漏： `transaction_id`、 `appid`、 `openid`、 `earn_points`、 `increased_points`、 `points_update_time`


* * *

### 268491269 — 该商圈未升级商圈会员 3.0

排查步骤：

1. 商圈会员 3.0 是新版会员体系，旧版商圈需要升级后才能使用 3.0 专属接口

2. 升级后需在小程序集成「商圈会员快速积分插件」2.4.0 及以上版本

3. 业务侧维护「brandid - 是否升级 3.0」配置，未升级商圈不调用 3.0 接口


* * *

### 269546507 — 商户证书序列号有误

排查步骤：

1. 商户平台 → 账户中心 → API 安全 → API 证书管理，确认当前生效证书序列号

2. `serial_no` 与签名所用私钥必须是同一对证书

3. 近期重新申请过证书的需更新本地配置

4. 多实例部署时确认所有节点使用同一份证书


* * *

### 269546525 — 错误的签名，导致验签失败

排查步骤：

1. 签名串拼接： `HTTP方法\nURL路径\n时间戳\n随机串\n请求体\n`

2. URL 必须用绝对路径，包含 query string，不含域名

3. GET 请求的请求体为空字符串（仍需要末尾的 `\n`）

4. 签名算法 SHA256withRSA，使用商户 API 证书私钥

5. `signature` 必须是标准 Base64 编码（非 URL-Safe）

6. 建议使用官方 SDK 自动签名


* * *

### 268435464 — 参数超出取值范围

排查步骤：

1. 看响应体 `detail.field` 定位超限字段

2. 注意同一字段在不同接口的长度限制可能不同：

   - `sub_mchid` 积分同步接口 ≤ 64 字符，待积分查询接口 ≤ 32 字符

   - `openid` 积分同步接口 ≤ 64 字符，待积分查询接口 ≤ 128 字符
3. 业务侧入参做长度校验


* * *

### 268490422 — 获取商户管理的商圈信息失败

排查步骤：

1. 确认当前 `mchid` 已开通智慧商圈产品权限

2. 商户平台 → 产品中心 → 智慧商圈，确认有关联的商圈与 brandid

3. 服务商需要确认有「服务商」身份 \+ 关联的品牌

4. 配置刚开通可能有数据同步延迟，等待几分钟后重试


* * *

### 272955916 — 订单已超过回传时效

排查步骤：

1. 必须在收到「商圈会员场内支付结果通知」当天内完成积分回传（0 点截止）

2. 业务侧的积分回传应实时处理，不要积压到次日

3. 失败重试机制必须在时效内完成，超过时效不要再重试

4. 超时未回传的订单无法补救

5. 业务侧可在每天 23:00 主动扫描当天未回传的订单做最后兜底


* * *

### 268490423 — 请求者非商圈服务商

排查步骤：

1. 确认调用方商户号在该 brandid 下是「服务商」身份

2. 商户平台 → 产品中心 → 智慧商圈 → 服务商资质，确认资质审核通过

3. 切换到正确的服务商商户号调用


* * *

### 268566836 — 插件验签时证书状态不可用

排查步骤：

1. 该错误指商户的微信支付平台证书不可用（过期 / 已撤销 / 未及时更新）

2. 调「获取平台证书」接口定期拉取最新平台证书，业务侧缓存多份覆盖轮换过渡期

3. 推荐使用官方 SDK 自动管理平台证书

4. 长期方案可迁移到微信支付公钥，规避证书轮换问题


* * *

### 268483843 — sub\_mchid 与 brandid 不匹配

排查步骤：

1. 确认 `sub_mchid` 已加入该 brandid 的商圈

2. 服务商平台 → 我的特约商户，确认该子商户已成功签约

3. 商圈 brandid 下需要把子商户号添加到「商圈商户列表」

4. 检查 `sub_mchid` 是否传错位、是否被截断


* * *

### 272955920 — 交易订单不属于此商圈，请检查订单信息

排查步骤：

1. `transaction_id` 必须是该商圈的「场内支付结果通知」中携带的微信订单号

2. 检查 `transaction_id` 是否传错（混淆了不同商圈的订单）

3. 服务商管理多个商圈时按 brandid 隔离订单数据

4. 普通微信支付订单（非商圈内交易）不能用此接口积分
